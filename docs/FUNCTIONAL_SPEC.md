# 手机 Web 动态验证码签到功能说明书

文档版本：v1.1  
状态：正式基线  
更新日期：2026-03-26  
项目：`wxapp-checkin`  
依据：`docs/REQUIREMENTS.md`、`docs/API_SPEC.md`

## 1. 目的

本文档描述手机 Web 版本的页面行为、角色能力、主链路交互与异常处理，供产品确认、联调、测试验收和后续实现拆分使用。

## 2. 当前定位

- 本文档描述的是当前正式 Web-only 产品行为。
- 仓库代码已删除历史小程序前端、Java 后端与独立改密链路。
- 如需追踪当前正式边界，以 `web` 分支现状和本文件为准。

## 3. 角色定义

- `normal`：普通签到人员
- `staff`：管理员 / 工作人员

## 4. 页面与功能

| 页面 | 访问角色 | 主要目标 | 核心动作 |
| --- | --- | --- | --- |
| `/login` | 全部 | 账号密码登录 | 输入学号与密码 |
| `/activities` | 已登录用户 | 浏览当前可见活动列表 | 查看活动、进入详情、staff 进入管理页 |
| `/activities/:id` | 已登录用户 | 查看单个活动详情 | 查看状态、跳转签到 / 签退页、staff 进入管理或名单页 |
| `/activities/:id/checkin` | `normal` | 输入签到码完成签到 | 输入 6 位码、提交、查看结果 |
| `/activities/:id/checkout` | `normal` | 输入签退码完成签退 | 输入 6 位码、提交、查看结果 |
| `/staff/activities/:id/manage` | `staff` | 展示动态码与活动管理 | 切换签到码 / 签退码、查看统计、一键全部签退 |
| `/staff/activities/:id/roster` | `staff` | 查看参会名单并修正状态 | 单人修正、批量修正、回前台刷新 |
| `/profile` | 已登录用户 | 查看本地会话资料与账户动作 | 查看姓名 / 学号 / 院系 / 社团、退出登录 |

## 5. 主链路说明

### 5.1 登录与进入业务态

1. 用户访问系统时先进入 `/login`。
2. 用户输入 `student_id + password`。
3. 后端查询 `suda_union` 完成账号校验、密码校验与会话签发。
4. 前端把 `session_token`、`role`、`permissions`、`user_profile` 写入本地会话。
5. 登录成功后进入 `/activities`。

### 5.2 普通用户活动浏览

- 普通用户只能看到本人已报名、已签到或已签退的活动。
- 活动列表默认分页加载，页面提供“加载更多”入口，避免一次性拉全量数据。
- 活动列表需展示：
  - 标题
  - 时间
  - 地点
  - 当前状态
  - 我的状态
- 进入详情页后，用户才能继续进入签到页或签退页。

### 5.3 活动详情

- 普通用户在详情页看到：
  - 活动名称、时间、地点、描述
  - 当前状态
  - 我的签到 / 签退时间
  - 当前是否允许签到 / 签退
- staff 在详情页看到：
  - 活动基础信息
  - 当前统计
  - “进入管理”
  - “参会名单”

### 5.4 普通用户签到

1. 用户进入具体活动的签到页。
2. 页面以大号数字输入控件展示 6 位输入框。
3. 用户输入管理员当前展示的签到码。
4. 提交后后端校验：
   - 会话有效
   - 用户已报名
   - 当前状态允许签到
   - 动态码正确且未过期
   - 当前时间片未重复提交
5. 成功后页面展示：
   - 活动标题
   - 动作类型
   - 成功提示
   - 服务器时间

### 5.5 普通用户签退

1. 用户进入具体活动的签退页。
2. 页面提示当前活动名称与动作“签退”。
3. 用户输入管理员展示的签退码。
4. 后端校验当前状态必须为“已签到未签退”。
5. 成功后页面展示签退成功结果，并返回活动页可见的最新状态。

### 5.6 管理员动态码管理页

1. 管理员进入活动管理页。
2. 页面默认展示当前动作对应的动态码。
3. 管理员可切换：
   - `checkin`
   - `checkout`
4. 页面必须持续展示：
   - 当前 6 位码
   - 剩余有效时间
   - 签到人数
   - 签退人数
5. 页面从后台恢复到前台时，必须立即刷新动态码与统计。
6. 浏览器若不支持自动保持屏幕常亮，页面要给出明确提示。

### 5.7 参会名单与状态修正

1. 管理员在详情页进入名单页。
2. 页面一次返回活动摘要与成员列表。
3. 管理员可以：
   - 单人修正签到 / 签退状态
   - 勾选多个成员后批量修正
4. 修正后页面必须刷新名单与统计，并清空已选成员。

### 5.8 一键全部签退

1. 管理员在活动管理页点击“一键全部签退”。
2. 页面必须先弹出确认提示。
3. 确认后，后端统一对“已签到未签退”用户执行批量签退。
4. 页面返回：
   - 影响人数
   - 批次号
   - 操作时间
5. 操作完成后，统计数字必须刷新。

## 6. 异常与恢复

### 6.1 会话失效

- 任何需要登录态的页面在收到 `session_expired` 后，必须清理本地会话并跳回 `/login`。

### 6.2 动态码错误或过期

- 错误码：明确提示“验证码错误，请重新确认”。
- 过期码：明确提示“验证码已过期，请重新输入最新验证码”。
- 重复提交：明确提示“当前时段已提交，请勿重复操作”。
- 错误次数过多（限流）：明确提示“验证码错误次数过多，请稍后再试”。

### 6.3 前后台切换

- 管理员页和普通用户输入页都必须监听 `visibilitychange`。
- 页面回到前台后必须重新校验会话、刷新时间差、拉取动态码或统计。

### 6.4 链接误入与权限不足

- 普通用户误入 staff 页面时，必须跳回活动列表。
- 用户进入当前状态不可执行的动作页时，页面必须明确提示“当前状态下暂不可执行该动作”。

## 7. 交互原则

- 只做手机优先布局。
- 6 位码输入必须适合单手操作。
- 成功 / 失败反馈必须在一屏内读完。
- 页面必须始终展示当前活动名称与当前动作，避免输错活动或输错码。
