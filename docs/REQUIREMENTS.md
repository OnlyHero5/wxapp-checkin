# 手机 Web 动态验证码签到系统需求文档

文档版本: v1.0
状态: 正式基线
更新日期: 2026-03-09
项目: `wxapp-checkin`

## 1. 文档目的

本文档定义 `wxapp-checkin` 从“微信小程序 + 动态二维码扫码”全面转向“手机浏览器 Web + 动态 6 位验证码输入”的目标需求。

本文件是当前正式需求基线，不再以微信小程序作为正式产品形态。实施阶段允许分阶段下线旧小程序代码，但最终目标是彻底剔除小程序相关正式业务逻辑。

## 2. 背景

当前仓库的历史实现曾以微信小程序为前端载体，依赖：

- `wx.login`
- 动态二维码展示
- 普通用户扫码提交
- 微信运行时与小程序发布流程

业务现已调整为：

- 管理员使用手机浏览器打开 Web 管理页
- 管理员按活动展示动态 6 位签到码或签退码
- 普通用户使用手机浏览器进入具体活动页面，在 7.5 秒内输入动态 6 位码完成签到或签退
- 管理员额外支持“当前活动一键全部签退”

同时，系统需要继续复用现有 `wxapp-checkin/backend` 与 `suda_union` 的数据能力，但禁止修改 `suda_union` 和 `suda-gs-ams` 的业务逻辑。

当前代码状态补充：

- `web/` 已是唯一正式前端。
- 后端正式入口统一为 `/api/web/**`。
- 历史小程序前端与旧微信登录正式入口已从仓库主干删除。

## 3. 建设目标

- 建成纯手机浏览器 Web 端签到/签退系统，替代微信小程序。
- 保留 `suda_union` 作为实名与报名事实源。
- 建立 `学号 + 姓名 + Passkey + 临时会话 + 浏览器唯一绑定` 的 Web 认证体系。
- 降低代签风险，但不引入短信、人脸识别等更重的身份能力。
- 兼容当前主流手机浏览器和主流手机屏幕尺寸。
- 最终删除小程序正式逻辑、扫码逻辑和二维码正式链路。

## 4. 范围

### 4.1 范围内

- 新手机 Web 前端
- 实名绑定、Passkey 注册与登录、浏览器绑定
- 活动列表、活动详情、输入签到码、输入签退码
- 管理员动态码展示页
- 管理员一键全部签退
- 管理员审核解绑
- 后端接口、扩展表、审计与测试重构
- `suda_union` 只读实名/报名校验
- 状态最终一致回写到 `suda_union`

### 4.2 范围外

- `suda-gs-ams` 业务逻辑改造
- `suda_union` 业务逻辑改造
- 微信小程序继续演进
- 桌面端后台系统
- 短信验证码、人脸识别、身份证 OCR 等新身份体系

## 5. 角色定义

- `normal`: 普通签到人员
- `staff`: 管理员/工作人员
- `review_admin`: 具备解绑审核权限的管理员；实现上可与 `staff` 合并

## 6. 核心业务规则

- 普通用户首次使用时，必须输入 `student_id + name`。
- 后端必须只读查询 `suda_union` 校验实名是否存在且一致。
- 首次实名校验通过后，必须完成 Passkey 注册。
- 后续登录时，必须做一次 Passkey 验证，登录成功后签发临时会话。
- 普通用户与管理员使用同一套登录流程。
- 管理员身份由后端基于 `suda_union` / 管理员名单判定，不由前端决定。
- 一个活跃浏览器绑定只能对应一个账号。
- 一个账号默认仅允许一个活跃浏览器绑定。
- 用户不得自助解绑或自助换绑浏览器。
- 更换浏览器或更换手机时，必须由管理员审核解绑后才能重新绑定。
- 普通用户必须先进入具体活动，再输入签到码或签退码。
- 报名资格以 `suda_union` 报名事实为准；未报名用户不得签到或签退。
- 动态码按“活动 + 动作 + 7.5 秒时间片”生成。
- 同一活动、同一动作、同一时间片内，所有管理员看到同一个 6 位动态码。
- 签到码和签退码彼此独立。
- 动态码只在 7.5 秒内有效，不保留当前二维码方案的宽限时间。
- 管理员可对当前活动执行“一键全部签退”。
- 一键全部签退只作用于“已签到未签退”的用户。
- 一键全部签退统一使用管理员点击时的服务端时间作为签退时间。
- 实施阶段允许分阶段下线小程序代码；最终必须删除小程序正式业务逻辑。

## 7. 功能需求

### 7.1 实名绑定与浏览器绑定

- `FR-001` 用户首次访问系统时，若无有效绑定，必须进入实名绑定流程。
- `FR-002` 实名绑定输入字段为 `student_id`、`name`。
- `FR-003` 后端必须只读查询 `suda_union` 校验实名是否真实存在。
- `FR-004` 未通过实名校验时，不允许进入 Passkey 注册或业务页面。
- `FR-005` 同一活跃浏览器绑定只能对应一个账号。
- `FR-006` 同一账号默认只允许一个活跃浏览器绑定。
- `FR-007` 用户不得自行解绑或自行更换绑定浏览器。
- `FR-008` 浏览器更换、手机更换必须通过管理员审核解绑后才能重新绑定。

### 7.2 Passkey 与登录

- `FR-009` 首次实名绑定成功后，必须完成 Passkey 注册。
- `FR-010` 未完成 Passkey 注册的用户不得进入业务页面。
- `FR-011` 后续登录必须通过 Passkey 验证一次。
- `FR-012` 登录成功后，后端签发 `session_token`。
- `FR-013` 会话失效后，用户必须重新登录。
- `FR-014` 管理员与普通用户复用同一套登录流程。
- `FR-015` 管理员权限必须由后端基于实名数据和权限规则判定。

### 7.3 活动列表与详情

- `FR-016` 普通用户只能看到本人已报名、已签到或已签退的活动。
- `FR-017` 管理员可以看到全部可管理活动。
- `FR-018` 活动详情必须展示标题、时间、地点、状态、统计等关键信息。
- `FR-019` 普通用户进入签到/签退页前，必须先进入具体活动。
- `FR-020` 未报名用户不得进入该活动的有效签到/签退成功路径。

### 7.4 管理员动态签到码与签退码

- `FR-021` 管理员在活动管理页可切换查看“签到码”和“签退码”。
- `FR-022` 动态码按“活动 + 动作 + 7.5 秒时间片”生成。
- `FR-023` 同一活动、同一动作、同一时间片内，所有管理员看到同一个 6 位码。
- `FR-024` 签到码与签退码必须独立。
- `FR-025` 管理员页必须展示当前动态码、剩余有效时间、签到人数、签退人数。
- `FR-026` 管理员页从后台切回前台时，必须立即刷新当前动态码与统计。

### 7.5 普通用户签到与签退

- `FR-027` 普通用户签到时，必须在具体活动签到页输入 6 位签到码。
- `FR-028` 普通用户签退时，必须在具体活动签退页输入 6 位签退码。
- `FR-029` 动态码仅在 7.5 秒内有效，过期后必须明确提示失效。
- `FR-030` 提交成功后，页面必须展示明确结果、动作类型与活动名称。
- `FR-031` 重复提交必须返回可识别的重复状态。
- `FR-032` 会话失效、未报名、状态不允许、码错误、码过期等场景必须返回清晰失败原因。

### 7.6 一键全部签退

- `FR-033` 管理员在活动管理页可执行“一键全部签退”。
- `FR-034` 一键全部签退只作用于“已签到且未签退”的用户。
- `FR-035` 一键全部签退统一使用管理员点击时的服务端时间作为签退时间。
- `FR-036` 一键全部签退执行后，活动统计必须更新。
- `FR-037` 一键全部签退必须写入管理员操作审计日志。
- `FR-038` 一键全部签退结果必须最终回写到 `suda_union`。

### 7.7 解绑审核

- `FR-039` 用户可发起解绑申请，但不得自行生效。
- `FR-040` 管理员可查看待审核解绑申请列表。
- `FR-041` 管理员可批准或拒绝解绑申请。
- `FR-042` 审核结果必须记录审核人、审核时间、审核意见。
- `FR-043` 审核批准后，原浏览器绑定失效，新浏览器才允许重新绑定。

### 7.8 审计与同步

- `FR-044` 所有签到、签退、一键全部签退都必须保留事件审计。
- `FR-045` 所有管理员高风险动作必须单独写管理员审计日志。
- `FR-046` 签到/签退状态必须最终一致回写到 `suda_union`。
- `FR-047` 系统不得要求修改 `suda_union` 业务逻辑才能完成新流程。

## 8. 非功能需求

- `NFR-001` 系统必须优先兼容主流手机浏览器和主流手机屏幕尺寸。
- `NFR-002` 页面必须以手机竖屏操作为主，关键操作支持单手完成。
- `NFR-003` 全站必须运行在 HTTPS。
- `NFR-004` 登录、动态码有效性和状态流转必须以后端时间为准。
- `NFR-005` 关键错误提示必须面向用户可理解，不能只返回技术异常。
- `NFR-006` 系统必须支持页面回前台自恢复。
- `NFR-007` 动态码生成与验码必须具备防猜测、防重放、限流能力。
- `NFR-008` 新设计必须允许最终彻底删除小程序正式逻辑。

## 9. 验收标准

- `AC-001` 未绑定用户首次访问时，只能进入实名绑定与 Passkey 注册流程。
- `AC-002` 输入错误的学号姓名时，不能绑定成功。
- `AC-003` 已绑定浏览器不能直接绑定第二个账号。
- `AC-004` 未经管理员审核，用户不能自行换浏览器重新绑定。
- `AC-005` Passkey 登录成功后，用户可进入活动列表。
- `AC-006` 普通用户只能看见本人相关活动。
- `AC-007` 管理员在同一活动同一时刻打开管理页时，看到相同的签到码或签退码。
- `AC-008` 普通用户在 7.5 秒内输入正确签到码可成功签到。
- `AC-009` 普通用户在 7.5 秒内输入正确签退码可成功签退。
- `AC-010` 动态码过期后，系统返回过期状态。
- `AC-011` 未报名用户输入正确动态码也不得签到成功。
- `AC-012` 一键全部签退后，仅“已签到未签退”用户被批量签退。
- `AC-013` 一键全部签退后，统计与记录正确更新。
- `AC-014` 解绑申请必须经过管理员审批后才生效。
- `AC-015` 在 iPhone Safari、iPhone Chrome/Edge、Android Chrome、Samsung Internet、微信内普通 H5 打开场景下，核心流程可正常完成。
