# iOS26 风格 Web 重塑设计

文档日期：2026-03-18  
适用范围：`wxapp-checkin/web`  
状态：已确认，可进入实现

## 1. 背景

`wxapp-checkin` 当前正式产品形态已经收口为手机浏览器 Web，前端位于 `web/`，技术栈为 `React + Vite + TypeScript + tdesign-mobile-react`。现有实现已经完成业务闭环，但视觉层仍偏“常规表单页 + 轻卡片”，和 iOS 26 所强调的层级、导航和控制感有明显差距。

本次目标不是做一套“炫技版玻璃 UI”，而是在不改变业务流、接口和权限边界的前提下，把高频页面重塑为更接近 iOS 26 的移动 Web 体验。

## 2. 设计依据

### 2.1 项目侧约束

- 当前正式前端仅允许围绕 `web/` 演进，不再按小程序设计。
- 页面必须手机优先，核心链路需在一屏内清晰完成。
- 必须优先复用现有共享层与 `tdesign-mobile-react`，避免页面分散造轮子。
- 动态 6 位码、统计、错误提示、一键签退都属于高频关键内容，可读性优先级高于视觉装饰。

### 2.2 Apple 官方线索

Apple 在 2025-06-09 发布的 iOS 26 预览中，将新的系统视觉语言明确归纳为 `Liquid Glass`。结合 Apple Developer 对 `Liquid Glass`、`Materials` 等文档的说明，可以提炼出两个和本项目强相关的结论：

- 玻璃材质更适合导航层、控制层、切换层和系统浮层。
- 正文内容区和关键信息区仍需保持足够的对比、清晰边界和稳定排版，不应被大面积模糊材质吞没。

## 3. 设计目标

### 3.1 视觉目标

- 建立“克制型 iOS 26 企业化”视觉基线。
- 让页面层级从“单张白卡片”升级为“顶部控制层 + 内容信息层 + 强焦点动作层”。
- 在登录、活动、签到码输入、管理员管理等高频页面上，明显提升质感、秩序感和移动端原生感。

### 3.2 实现目标

- 尽量通过共享层重构实现整体升级，而不是逐页堆样式补丁。
- 最大化复用 `tdesign-mobile-react` 的 `Button / Tabs / Tag / Cell / NoticeBar` 等现有原语。
- 不改变业务路由、数据流、会话逻辑和接口契约。

## 4. 总体方案

本次采用“内容实体化、控制轻玻璃化”的混合方案：

- 页面背景：改为更接近 iOS 系统纸感的分层浅灰背景。
- 顶部壳层：引入轻玻璃头部，承担页面标题、上下文和次级动作承载。
- 内容卡片：保持实体卡片与清晰边界，不做满屏玻璃。
- 控制组件：按钮、Tabs、状态提示、动态码主区允许使用轻度模糊、半透明、高光边界等材质语言。
- 动效：只保留 150ms 到 250ms 的轻量过渡，不做夸张位移和漂浮动画。

## 5. 视觉语言

### 5.1 色彩

采用浅色系统感基线，避免历史 Moonshot 深色方案直接迁移：

- 页面底色：偏系统灰白，分出根背景和卡片背景两层。
- 主文字：高对比深灰黑。
- 次文字：中灰，保留可读性，不走发蓝发虚路线。
- 强调色：以接近 iOS 系统蓝的单一主色为主，用于主按钮、焦点、链接、选中态。
- 成功 / 警告 / 错误：继续沿用组件库语义色，不另造彩色体系。

### 5.2 圆角与阴影

- 内容卡片圆角控制在 `18px` 左右，接近系统卡片语言，但不夸张。
- 按钮与输入控件统一更圆润，主按钮圆角可高于内容卡片。
- 阴影强调“层级识别”而非“漂浮感”，以轻阴影和 1px 高光边界为主。

### 5.3 材质

- 玻璃材质只用于：页头、分段切换容器、动态码舱、轻提示容器。
- 内容列表、活动信息、统计信息仍使用实体化卡片或分组面板。
- 模糊强度必须克制，防止文字发虚和低端设备卡顿。

### 5.4 字体与数字

- 中文继续优先 `PingFang SC` 风格系统栈。
- 数字类信息统一使用 `tabular-nums`，保证 6 位码、剩余秒数和统计对齐。
- 动态码和码输入区需要更强字重、更大字号和更稳定的间距。

## 6. 组件级改造

### 6.1 `MobilePage`

从“单张居中卡片”升级为：

- 页面背景层
- 顶部玻璃头部层
- 主内容堆叠层

新职责：

- 处理安全区留白
- 承担标题、eyebrow、返回区域、可选头部操作区
- 为业务页面提供统一的手机纵向阅读节奏

### 6.2 `ActivityMetaPanel`

从“信息段落合集”升级为“摘要卡”：

- 头部：标题、副标题、状态标签
- 主体：时间、地点、描述、我的状态、当前状态
- 统计区：以更强的视觉分组呈现“应到 / 累计签到 / 已签退 / 未签退”
- 底部：文本链接或补充动作区

### 6.3 `AppButton`

统一成两层语义：

- `primary`：高对比实体按钮，承担推进主流程
- `secondary`：轻材质或描边按钮，承担切换、重试、次要动作

要求：

- 高度满足移动触控区
- 保留 TDesign 原生交互与 loading 态
- 通过项目级 class 收口圆角、阴影、按压反馈

### 6.4 `CodeInput`

输入逻辑不变，但视觉改造为更强的六码输入体验：

- 继续保留真实单 `input`
- 视觉上通过字距、背景分段、投影边界提升“六码格”感
- 明确当前动作文案，减少误输场景

### 6.5 `DynamicCodePanel`

这是本次重塑的核心共享组件：

- 顶部使用分段切换承载“签到码 / 签退码”
- 中部动态码做成玻璃舱，强调当前码和剩余秒数
- 下方统计信息分层为更易扫读的面板
- 刷新动作与批量签退动作视觉上分离

### 6.6 `InlineNotice` 与 `StatusTag`

- `InlineNotice` 继续使用组件库 `NoticeBar`，但通过容器和圆角提升系统感
- `StatusTag` 保留组件库 `Tag`，统一色彩、尺寸和轻量胶囊效果

## 7. 页面落地策略

### 7.1 登录页 / 改密页

- 采用更接近系统登录卡片的结构
- 减少解释性文字的存在感
- 突出输入和主按钮

### 7.2 活动列表页

- 分组标题更像系统列表分段标题
- 活动卡片更强调摘要与状态
- 管理入口与详情入口保持统一的文本动作风格

### 7.3 活动详情页

- 详情卡置顶
- 动作按钮更强区分“主动作 / 次动作 / 管理动作”
- 让用户更快判断当前是否还能签到或签退

### 7.4 签到 / 签退页

- 活动名、动作类型、码输入区尽量一屏可见
- 成功 / 失败提示稳定停留在输入区附近
- 码输入组件成为页面视觉中心

### 7.5 管理员页

- 作为本次改造的重点页面
- 顶部活动摘要 + 中部动态码舱 + 下部统计区 + 底部批量动作区
- 支持从后台切回前台后继续稳定刷新，不依赖动画强调

## 8. 动效与可访问性

- 过渡只用于背景、阴影、透明度、轻微缩放
- 默认时长控制在 `150ms-250ms`
- `prefers-reduced-motion` 下关闭非必要过渡
- 所有点击区保持至少 `44px`
- 保证正文、说明、状态和统计文案满足基础对比度

## 9. 非目标

以下内容不在本次设计范围：

- 改动业务路由、权限、接口或数据结构
- 引入新的全局状态库
- 切换组件库
- 做桌面端后台化布局
- 追求满屏 Liquid Glass 或演示型视觉特效

## 10. 验证原则

- 共享层升级后，登录、活动列表、活动详情、签到/签退、管理员页都必须保持主流程不变
- 单测需覆盖共享层变动影响到的页面结构和主要交互
- 最终至少执行：
  - `cd /home/psx/app/wxapp-checkin/web && npm test`
  - `cd /home/psx/app/wxapp-checkin/web && npm run build`

## 11. 实施建议

优先顺序如下：

1. 先重做 `base.css` 与共享层组件，建立新视觉基线
2. 再改登录、列表、详情、输入、管理员等页面组合
3. 最后统一修测试、构建和细节回归

这样可以用最少的页面级改动，换取最大范围的整体质感提升。
