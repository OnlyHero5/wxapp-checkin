# Web 导航信息架构与个人中心设计

文档日期：2026-03-18  
适用范围：`wxapp-checkin/web`  
状态：已确认，可进入实现

## 1. 背景

`wxapp-checkin` 当前正式产品已经收口为手机 Web，正式业务基线来自：

- `docs/REQUIREMENTS.md`
- `docs/FUNCTIONAL_SPEC.md`
- `docs/API_SPEC.md`

这些文档定义的主链路一直是：

- 登录 / 改密
- 活动列表
- 活动详情
- 具体活动下的签到 / 签退
- 管理员在具体活动下的发码管理 / 名单修正

也就是说，系统真正的业务对象是“活动”和“当前登录用户”，而不是“进行中”“历史活动”“活动详情”这类页面状态。

但当前 Web 实现把底部导航混成了三种完全不同的职责：

- 活动列表页拿底部栏做页内锚点跳转
- 详情页 / 签到页 / 管理页拿底部栏做返回链路
- 组件层允许把锚点与路由按钮放进同一套视觉容器

这会让用户误以为“进行中 / 历史活动 / 活动详情”是同一层级导航对象，直接偏离正式需求。

## 2. 设计输入

### 2.1 项目内已确认事实

- `docs/FUNCTIONAL_SPEC.md` 已明确页面结构是 `/activities -> /activities/:id -> /activities/:id/checkin|checkout`。
- `docs/REQUIREMENTS.md` 已明确普通用户必须先进入具体活动，再输入签到码或签退码。
- `web/src/app/router.tsx` 当前也以 `/activities` 作为登录后的业务首页。
- 历史“小程序 TabBar + 个人中心”方案仅属于旧阶段，不再是正式 Web 基线。

### 2.2 外部导航原则

这次分析参考了以下当前可公开获取的导航模式资料：

- Apple Bars 文档：tab bar 用于同级视图切换，不用于返回链路
- Android Navigation 模式文档：navigation bar 应承载稳定的 top-level destinations，二级内容应用页内导航
- Eventbrite / Whova / EventMobi 这类活动产品：顶级入口通常是活动域、票券域、账户域，不会把“详情页”做成顶级 tab

由此可得出一个和本项目高度一致的原则：

- 顶级导航只承载稳定、跨页面存在、用户可预期的对象域
- 页内分段、返回路径、动作页状态，不应冒充顶级导航

## 3. 当前问题归纳

### 3.1 结构问题

当前 `PageBottomNav` 的职责被混用了：

- 顶级导航
- 二级筛选
- 返回路径

这会导致同一套视觉控件同时表达“去别的一级页面”“回上一层”“跳本页锚点”三种语义，信息层级失真。

### 3.2 业务问题

从业务角度看：

- “进行中 / 历史活动”只是活动列表的分组方式
- “活动详情 / 活动签到 / 活动签退 / 活动管理 / 参会名单”都是某个活动下的子页面
- 真正独立的全局对象只有“活动”和“当前用户”

因此现在的导航不是“导航到了正确对象”，而是在“导航到页面实现细节”。

### 3.3 维护问题

继续沿用当前混合底部栏，会产生三类长期成本：

- 新增个人中心时，不知道该塞成一级页还是工具页
- staff 页面越来越多后，活动域内部层级会继续失控
- 页面测试会把“底部栏文案”误当成信息架构，而不是把路由语义写清楚

## 4. 方案比较

### 方案 A：保留现有混合底部栏，只微调文案

优点：

- 改动最小
- 不需要调整路由壳层

缺点：

- 错误不在文案，而在层级
- “进行中 / 历史活动 / 活动详情”仍会继续冒充一级导航
- 后续个人中心仍然无处安放

### 方案 B：顶级导航收口为“活动 / 我的”，活动内再做二级分段

优点：

- 与正式需求和当前路由完全一致
- 顶级对象稳定，用户心智简单
- 活动域内部可以继续扩展详情、发码、名单，而不污染全局导航
- 能自然补出个人中心

缺点：

- 需要重构当前底部栏职责
- 需要新增个人中心页面和测试

### 方案 C：按角色拆顶级导航，普通用户“活动 / 我的”，staff“活动 / 工作台 / 我的”

优点：

- staff 高频管理动作更集中
- 后续若出现跨活动待办，可更快落地

缺点：

- 两套导航心智不一致
- 当前正式文档并未定义独立 staff 工作台
- 第一版容易把范围做大

### 结论

采用 **方案 B**：

- 顶级导航固定为 `活动`、`我的`
- `进行中 / 历史活动` 下沉为活动列表页内二级分段
- 活动详情、签到、签退、管理、名单都归属到 `活动` 域内
- 暂不新增独立 `工作台` 顶级入口

## 5. 推荐信息架构

### 5.1 顶级导航

顶级导航只保留两个入口：

1. `活动`
2. `我的`

激活态映射规则：

- 以下路径统一归属 `活动`
  - `/activities`
  - `/activities/:id`
  - `/activities/:id/checkin`
  - `/activities/:id/checkout`
  - `/staff/activities/:id/manage`
  - `/staff/activities/:id/roster`
- 以下路径归属 `我的`
  - `/profile`

`/login` 与 `/change-password` 不显示业务态顶级导航。

### 5.2 活动域

`活动` 是业务主域，也是登录后的默认落点。

内容结构：

- 顶部页面标题与角色说明
- 活动列表页内二级分段
  - `进行中`
  - `历史活动`
- 活动卡片
  - 标题
  - 时间
  - 地点
  - 当前状态
  - 我的状态 / 管理统计

进入活动详情后，继续在活动域内部完成所有操作：

- 普通用户
  - 查看活动详情
  - 去签到
  - 去签退
- staff
  - 进入管理
  - 参会名单

### 5.3 我的域

`我的` 用来承接“当前登录用户”这个全局对象，不承接活动子页面。

第一阶段建议内容：

- 基本身份信息
  - 姓名
  - 学号
  - 部门
  - 社团
- 当前角色信息
  - 普通用户 / 工作人员
  - 权限摘要
- 账户动作
  - 修改密码
  - 退出登录

说明：

- 当前登录态已经持久化了 `user_profile / role / permissions`
- 后端 DTO 已预留更多个人资料字段，但本轮不强行扩范围
- 如果后续确认要展示积分、头像或更完整档案，再在 `我的` 域内扩展，不进入 `活动` 顶级导航

## 6. 页面级导航规则

### 6.1 活动列表页

- 不再使用底部栏做 `进行中 / 历史活动` 锚点
- 改为页内二级导航
- 推荐使用更贴近内容区的 segmented control / tabs

### 6.2 活动详情与动作页

- 不再把“活动列表 / 活动详情”放进底部 tab
- 这些页面应使用页头返回或清晰文本返回入口
- 顶级导航仍显示 `活动` 为当前域，但不把子页面平铺成 tab

### 6.3 管理页与名单页

- 仍属于 `活动` 域
- 保持 `活动` 顶级入口激活
- 页面内部通过标题、返回和按钮区分“发码管理”与“名单修正”

## 7. UI 结构建议

### 7.1 全局业务壳层

建议把顶级导航提升为“业务态全局壳层”的一部分，而不是让每个页面自己决定底部栏内容。

职责：

- 统一控制哪些页面显示顶级导航
- 统一计算当前激活入口
- 避免每个页面复制导航文案和激活态规则

### 7.2 二级导航与返回

导航控件按层级拆开：

- 顶级导航：`活动 / 我的`
- 二级导航：活动列表页内的 `进行中 / 历史活动`
- 返回链路：详情页和动作页的页头返回

不再复用同一个组件表达三种语义。

## 8. 非目标

以下内容不在本次设计范围：

- 新增 staff 独立工作台首页
- 改动活动业务规则、权限规则、接口契约
- 把“我的”扩展成复杂档案系统
- 恢复历史小程序式多 tab 结构

## 9. 验证原则

- 登录后默认仍进入 `/activities`
- 任一活动子路由都应保持 `活动` 顶级入口高亮
- `进行中 / 历史活动` 只能作为活动页内分段，不再作为顶级 tab
- `我的` 页必须能展示当前用户基本身份，并提供账户动作
- 现有签到、签退、发码、名单路径不改变业务含义

## 10. 实施建议

建议按以下顺序推进：

1. 先落全局业务壳层与顶级导航组件
2. 再移除活动域页面上对混合 `PageBottomNav` 的滥用
3. 接着新增 `我的` 页与路由
4. 最后把活动列表页的 `进行中 / 历史活动` 改成页内二级导航

这样可以先把层级纠正，再补页面内容，避免边改边继续混层。
