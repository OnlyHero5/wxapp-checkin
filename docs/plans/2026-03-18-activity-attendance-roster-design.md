# 活动参会名单与管理员手工修正设计

## 1. 背景

当前 `wxapp-checkin` 的 Web 管理员能力已经覆盖：

- 活动详情查看
- 动态签到码 / 签退码展示
- 一键全部签退

但管理员仍缺少“按活动查看参会名单并手工修正签到/签退状态”的能力。实际运营中会出现以下高频场景：

- 某位同学线下已到场，但未及时输入签到码，需要管理员补签到
- 某位同学误签到，需要管理员撤销签到
- 某位同学已离场但未输入签退码，需要管理员补签退
- 某位同学签退记录有误，需要管理员撤销签退
- 某一批同学需要一次性批量修正状态

本设计目标是在不破坏现有 Web-only 活动、动态码、统计与 legacy 回写主干的前提下，补齐管理员的名单查询与单个 / 批量状态修正能力。

## 2. 已确认业务规则

### 2.1 页面入口

- 管理员在活动详情页看到新的“参会名单”按钮
- 点击后进入独立页面 `/staff/activities/:id/roster`
- 不把参会名单直接塞进现有动态码管理页，避免单页职责过重

### 2.2 名单范围与展示字段

- 名单范围默认是“当前活动所有已报名人员”
- 每行展示以下字段：
  - 姓名
  - 学号
  - 签到状态
  - 签退状态

### 2.3 修改口径

本需求的 UI 语义是“签到”“签退”两个事项独立展示，但数据约束不是完全自由组合，而是遵循以下业务联动：

- 把“签到”设为未签时，必须自动把“签退”也设为未签
- 把“签退”设为已签退时，必须自动把“签到”补成已签
- 禁止出现“未签到但已签退”的脏状态

因此前端展示是“两项状态”，后端落库仍统一收敛为现有三态：

- `none`：未签到未签退
- `checked_in`：已签到未签退
- `checked_out`：已签到已签退

### 2.4 批量交互

- 管理员先勾选若干参会人
- 再执行批量动作：
  - 设为已签到
  - 设为未签到
  - 设为已签退
  - 设为未签退
- 批量修改与单个修改走同一套后端状态规则与审计逻辑

## 3. 方案比较

### 方案 A：把参会名单直接放进现有活动管理页

优点：

- 表面上页面数量更少
- 路由增量较小

缺点：

- 现有 `/staff/activities/:id/manage` 已承担动态码、剩余时间、统计、一键全部签退
- 再塞名单和批量修正后，移动端页面会过重
- 发码与名单修正属于两类不同任务，混放会提高误操作率
- 不符合本次确认的“活动详情 -> 参会名单独立页”入口诉求

### 方案 B：新增独立参会名单页，单个 / 批量修改共用统一修正接口

优点：

- 入口清晰，符合当前产品诉求
- 页面职责单一，便于手机端使用
- 单个修改和批量修改复用同一状态机、审计与同步链路
- 对现有动态码页侵入最小

缺点：

- 需要新增一条 staff 路由、一组前后端 DTO / API / 页面

### 方案 C：新增名单页，但单个修改与批量修改分别做两套写接口

优点：

- 单行交互可以做得更“就地”

缺点：

- 后端状态变更口径分叉
- 统计、审计、legacy 回写容易出现两套逻辑
- 长期维护成本更高

### 结论

采用 **方案 B**：

- 新增独立参会名单页
- 单个 / 批量修改共用统一管理员状态修正服务

## 4. 页面与交互设计

### 4.1 活动详情页

管理员视角下，活动详情页保留现有“进入管理”按钮，并新增“参会名单”按钮。

职责区分如下：

- “进入管理”：继续处理动态码、实时统计、一键全部签退
- “参会名单”：进入名单查询与状态修正页

这样可以让“发码”和“修状态”两个高频动作分离，减少单页复杂度。

### 4.2 参会名单页结构

页面建议分为三块：

1. 活动摘要区
- 复用现有 `ActivityMetaPanel`
- 展示活动名称、时间、地点、应到人数、已签到人数、已签退人数
- 让管理员始终知道当前正在操作哪一场活动

2. 批量操作区
- 展示当前已选人数
- 提供四个批量动作：
  - 设为已签到
  - 设为未签到
  - 设为已签退
  - 设为未签退

3. 名单列表区
- 每行展示：姓名、学号、签到状态、签退状态
- 每行带勾选框，用于批量操作
- 每行同时提供单人快捷修正动作

### 4.3 单人操作

每行支持以下单人修正：

- 签到 -> 已签 / 未签
- 签退 -> 已签退 / 未签退

前端只暴露合法动作，不允许用户自己拼非法组合。

例如：

- 当前未签到未签退：
  - 可设为已签到
  - 可设为已签退（后端自动补签到）
- 当前已签到未签退：
  - 可设为未签到（后端同时清签退）
  - 可设为已签退
- 当前已签到已签退：
  - 可设为未签退
  - 可设为未签到（后端同时清签到）

### 4.4 批量操作确认

批量动作执行前弹确认框，明确说明联动规则：

- 设为未签到会同时清空签退
- 设为已签退会自动补成已签到

这样可以降低误判和误触后的争议。

### 4.5 刷新策略

名单页在以下时机重新拉取活动摘要与名单：

- 页面首次进入
- 单个修改成功后
- 批量修改成功后
- 页面回到前台时（`visibilitychange`）

这样可以降低多个管理员同时操作时看到旧状态的窗口。

## 5. 后端接口设计

## 5.1 名单查询接口

新增 staff 接口：

`GET /api/web/staff/activities/{activityId}/roster`

返回：

- 活动基础信息
- 当前统计信息
- 已报名人员名单
- 每个成员的：
  - `user_id`
  - `name`
  - `student_id`
  - `checked_in`
  - `checked_out`
  - 当前显示用签到时间
  - 当前显示用签退时间

名单只返回当前活动“已报名人员”，不返回未报名用户，避免管理员误修改无报名关系的数据。

## 5.2 状态修正接口

新增统一写接口：

`POST /api/web/staff/activities/{activityId}/attendance-adjustments`

请求体示意：

```json
{
  "user_ids": [12, 18, 25],
  "patch": {
    "checked_in": false,
    "checked_out": false
  },
  "reason": "管理员手工修正"
}
```

设计约束：

- `user_ids` 支持单个或多个
- `patch` 至少包含一个目标字段
- 服务端统一把 patch 收敛为合法最终状态
- 不拆“单个接口”和“批量接口”两套写模型

### 5.2.1 patch 到最终状态的归一规则

服务端统一按以下规则决定最终状态：

- `checked_in = false`
  - 最终状态：`none`
  - 同时强制 `checked_out = false`

- `checked_out = true`
  - 最终状态：`checked_out`
  - 同时强制 `checked_in = true`

- `checked_in = true` 且 `checked_out = false`
  - 最终状态：`checked_in`

- `checked_out = false` 且当前 / 目标仍为已签到
  - 最终状态：`checked_in`

因此外部传入的 patch 只是“意图”，最终状态由服务端统一归一，避免前端各自实现状态机。

### 5.2.2 权限与参数校验

接口必须校验：

- 当前会话必须是 `staff`
- 活动存在且可管理
- `user_ids` 非空
- 所有目标用户都属于该活动且已报名
- `patch` 不能是空补丁

若存在不属于活动的用户，接口应整体失败，避免“只成功一半却没人察觉”的风险。

## 6. 数据与同步设计

### 6.1 主状态模型不变

本次不修改 `wx_user_activity_status.status` 的三态模型：

- `none`
- `checked_in`
- `checked_out`

原因：

- 现有活动列表、详情、动态码消费、批量签退都依赖这一模型
- 若直接改成双布尔落库，会扩大改动面并增加回归风险
- 本需求只是管理端展示成两项状态，不要求重构核心状态域模型

### 6.2 统计更新

手工修正时，必须基于“旧状态 -> 新状态”的转换，计算统计差量并原子更新：

- `none -> checked_in`：`checkin_count +1`
- `none -> checked_out`：`checkout_count +1`
- `checked_in -> none`：`checkin_count -1`
- `checked_in -> checked_out`：`checkin_count -1`、`checkout_count +1`
- `checked_out -> checked_in`：`checkin_count +1`、`checkout_count -1`
- `checked_out -> none`：`checkout_count -1`

这里的统计口径继续保持当前系统定义：

- `checkin_count`：已签到未签退人数
- `checkout_count`：已签退人数

页面展示“累计签到人数”时，仍由前端使用 `checkin_count + checkout_count` 推导。

### 6.3 审计日志

所有管理员名单修正都必须写入 `web_admin_audit_log`。

审计内容至少包含：

- 操作者 user_id
- activity_id
- 批次 id
- user_ids
- 修改原因 `reason`
- 每个目标用户的旧状态 / 新状态
- 服务端时间
- 操作类型（例如 `attendance_adjustment`）

无论是单个还是批量，都走同一审计模型。

### 6.4 legacy 回写

本次需求不能只改扩展库本地状态，否则会出现 Web 端和 legacy 管理域长期不一致。

因此每次名单修正后，都必须生成 outbox 事件并由 `OutboxRelayService` 回写 legacy。

推荐做法：

- 保留现有 outbox 主链路
- 新增“管理员状态修正”事件类型 / payload 语义
- payload 直接携带目标 `check_in` / `check_out` 值，而不是复用当前“正向签到 / 正向签退”语义

原因：

- 当前 `wx_checkin_event` / outbox 更适合表达“发生了一次签到 / 签退”
- 但管理员修正场景存在“撤销签到”“撤销签退”
- 若仍硬塞到 `checkin` / `checkout` 事件里，会导致语义扭曲

目标回写映射如下：

- `none` -> `check_in = 0`, `check_out = 0`
- `checked_in` -> `check_in = 1`, `check_out = 0`
- `checked_out` -> `check_in = 1`, `check_out = 1`

这样才能完整覆盖：

- 已签改未签
- 未签改已签
- 已签退改未签退
- 未签退改已签退

### 6.5 时间展示口径

活动详情与名单页中的“签到时间 / 签退时间”应当以当前状态为准做展示裁剪：

- 当前状态为 `none`
  - 不显示签到时间
  - 不显示签退时间
- 当前状态为 `checked_in`
  - 显示签到时间
  - 不显示签退时间
- 当前状态为 `checked_out`
  - 显示签到时间
  - 显示签退时间

这样可以避免“状态已被撤销，但页面还展示旧时间”的误导。

## 7. 前端实现边界

### 7.1 允许复用的现有能力

- `ActivityMetaPanel`
- `MobilePage`
- `PageBottomNav`
- 现有会话跳转策略（会话过期 / 强制改密）
- `subscribePageVisible`
- `AppButton`

### 7.2 新增前端模块

建议新增：

- `web/src/pages/activity-roster/ActivityRosterPage.tsx`
- `web/src/pages/activity-roster/ActivityRosterPage.test.tsx`
- `web/src/features/staff/roster-api.ts` 或在现有 `features/staff/api.ts` 中扩展名单相关契约
- 若名单行交互较复杂，可新增 `features/staff/components/AttendanceRosterList.tsx`

### 7.3 样式策略

遵循仓库约束：

- 优先使用现有组件库能力
- 尽量复用现有共享组件和 token
- 仅在组件库无法覆盖名单页布局时补最小必要 CSS
- 避免页面级散落样式

## 8. 测试设计

### 8.1 后端测试

必须新增 / 覆盖以下测试：

- 名单查询只返回已报名人员
- 单个用户：
  - 未签到 -> 已签到
  - 已签到 -> 未签到
  - 已签到 -> 已签退
  - 已签退 -> 未签退
  - 已签退 -> 未签到
- 批量用户混合状态修正时：
  - 最终状态正确
  - 统计差量正确
  - 审计正确
  - outbox payload 正确
- 非 staff 访问失败
- 活动不存在失败
- `user_ids` 为空失败
- 目标用户不属于活动失败
- 非法 patch 失败

### 8.2 前端测试

必须新增 / 覆盖以下测试：

- 活动详情 staff 视角出现“参会名单”按钮
- 点击后可进入名单页
- 名单页能正确渲染活动摘要与名单
- 单人修改成功后刷新列表与统计
- 批量修改成功后刷新列表与统计
- 批量操作确认框出现正确文案
- 页面回到前台时自动刷新
- 会话过期 / 强制改密跳转保持一致

## 9. 风险与控制

### 风险 1：管理员修正与定时 legacy pull 打架

若本地状态更新后，legacy 尚未完成回写，而定时 pull 又先读到旧值，可能造成短时回退。

控制：

- 名单修正沿用 outbox 最终一致性链路
- 与现有相关回归测试一起补充“管理员修正后的同步时序”测试
- 前端在修改成功后主动刷新，必要时可以预留短轮询能力

### 风险 2：统计口径被错误增减

名单修正不是单方向状态推进，若直接写死增减，很容易把统计改坏。

控制：

- 服务端必须基于“旧状态 -> 新状态”计算 delta
- 测试覆盖所有状态转换组合

### 风险 3：撤销签到 / 签退与旧事件模型语义冲突

控制：

- 不把管理员撤销动作硬塞成普通 `checkin` / `checkout` 事件语义
- 对 legacy 回写采用目标状态快照型 payload
- 审计和同步语义分层，不混用

## 10. 最终结论

本需求采用以下实现基线：

- 在活动详情页新增“参会名单”入口
- 新增独立 staff 参会名单页
- 名单范围为当前活动所有已报名人员
- 单个与批量状态修正共用统一后端接口
- 前端展示为“签到 / 签退”两项状态
- 后端继续落库到现有三态模型
- 所有修正必须同时覆盖：
  - 本地状态
  - 活动统计
  - 管理员审计
  - outbox 回写 legacy

该方案对现有 Web-only 主干侵入最小，同时能满足管理员对单个 / 批量修正签到签退状态的核心诉求。
