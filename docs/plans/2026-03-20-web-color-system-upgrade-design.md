# 手机 Web 业务态增色设计

文档日期：2026-03-20  
适用范围：`wxapp-checkin/web`  
状态：已确认，可进入实现

## 1. 背景

`wxapp-checkin/web` 已完成手机 Web 主链路建设，当前在登录、活动列表、详情、签到、签退、工作人员管理与名单页之间已经具备稳定的信息架构与共享组件体系。现状问题不在于结构混乱，而在于全局视觉 token 过于保守，导致以下体验不够明显：

- 用户进入页面后，难以仅靠视觉快速判断当前是“签到”“签退”还是“管理”场景。
- 活动卡片、动态码舱、统计区、底部导航和主按钮之间的层级差异主要依赖留白与边框，色彩参与度不足。
- 登录页、列表页和业务动作页虽然结构统一，但在“产品识别度”和“业务动作确认感”上仍偏平。

本次设计不重做路由、信息架构和组件库选型，而是在现有共享层基础上补齐“业务态增色系统”。

## 2. 设计目标

### 2.1 视觉目标

- 保留当前浅色、清晰、手机优先的产品基线，不改成深色产品。
- 让签到、签退、管理三类高频业务态拥有一眼可辨的色彩识别。
- 提升活动卡、动态码、统计卡、页头与按钮的视觉聚焦感，而不牺牲正文可读性。

### 2.2 实现目标

- 尽量通过 `base.css` token 和共享组件扩展完成全站升级，减少逐页散落样式补丁。
- 优先复用 `tdesign-mobile-react` 的 `Button / Tag / NoticeBar / Tabs / CellGroup` 能力，不新引入 UI 库。
- 让登录页、列表页、详情页、签到页、签退页、管理页、名单页、个人中心都能进入同一套色彩语言。

## 3. 外部依据

### 3.1 Apple 2025 新设计语言

Apple 在 2025-06-09 发布的新软件设计说明与 `Liquid Glass` 文档中，明确把颜色与玻璃材质更多用于导航、控制、切换和系统浮层，而不是把正文信息区整体做成高噪声视觉层。这和当前项目的需求高度一致：用户在手机端需要快速识别动作，而不是被装饰性效果分散注意力。

参考资料：

- Apple Newsroom, 2025-06-09  
  https://www.apple.com/newsroom/2025/06/apple-introduces-a-delightful-and-elegant-new-software-design/
- Apple Developer, `Liquid Glass`  
  https://developer.apple.com/documentation/technologyoverviews/liquid-glass

### 3.2 Material 3 色彩角色

Material 3 的 `color roles` 强调：界面不应只依赖一个主色，而应在稳定的 `surface` 体系上，用不同角色色建立层级、状态和动作识别。对于本项目，这意味着不应把全站统一刷成同一种品牌色，而应引入“默认品牌色 + 业务态角色色”的双层体系。

参考资料：

- Material 3 Color Roles  
  https://m3.material.io/styles/color/roles

## 4. 总体方案

本次采用“中性内容层 + 业务态强调层”的全站方案：

- 页面背景、正文容器、详情区、列表区保持浅中性层，确保连续阅读稳定。
- 页头、主按钮、选中态、动态码舱、统计块、提示条、底部导航活动态承担主要的色彩表达。
- 不做满屏彩色渐变，不把所有卡片都染色，而是只给“当前动作相关的控制层和摘要层”加色。
- 通过页面级 tone 与组件级 tone 两层收口，让不同页面能共享一套色彩系统。

## 5. 色彩系统

### 5.1 中性基底

中性层继续承担产品的“纸感”和“可读性地基”：

- 根背景：浅灰白，维持系统感与轻层次。
- 卡片背景：白色到微暖白，不做明显彩色底。
- 边框：浅灰蓝系中性边框，保证信息分组。
- 正文：高对比深灰，弱化发蓝发虚的文字观感。

### 5.2 默认品牌层

默认品牌层服务于非业务态页面和通用组件，包括：

- 登录页
- 改密页
- 个人中心
- 活动列表的默认主按钮和链接
- 全站默认 focus ring 与链接色

该层选择偏蓝青的清爽品牌色，但不会与签到/签退/管理态争夺识别权。

### 5.3 业务态角色色

本次引入三类业务态颜色：

- `checkin`：青绿 / 绿松石系
  传达“进入、到场、通过”的即时确认感。
- `checkout`：琥珀 / 橙金系
  传达“离场、完成、收口”的动作感，且与签到态形成明确对照。
- `staff`：蓝紫 / 靛紫系
  传达“控制台、发码、统计、管理”的系统控制感。

三类业务态颜色仅用于：

- 页头辅助氛围
- 主按钮和次按钮差异
- 动态码主舱
- 统计卡重点数字
- Tabs / Tag / NoticeBar 的业务态变体
- 码输入框、焦点边框和结果态提示

### 5.4 语义提示色

成功、警告、错误继续以组件库语义色为基础，不再自造一套危险色体系；本次只做容器层和边框层收口，让提示样式更融入整体。

## 6. 页面策略

### 6.1 登录页 / 改密页

- 使用默认品牌层，不强行套入签到/签退色。
- 页头和主按钮增强品牌识别，但表单仍保持高对比与低噪声。
- 让“主动作按钮”和错误提示更有存在感，避免整体像一张灰白表单。

### 6.2 活动列表页

- 列表仍以中性内容卡为主。
- 分段导航、状态标签、链接和活动卡摘要加入轻色阶提示。
- 对工作人员列表，在“进入管理”入口和统计摘要中更明显地引入管理态色彩。

### 6.3 活动详情页

- 作为动作闸门页，重点强化“去签到 / 去签退 / 进入管理”按钮的差异。
- 用户态下，签到入口优先使用签到色，签退入口优先使用签退色。
- 工作人员态下，管理入口用管理色，名单入口保持次按钮。

### 6.4 签到页

- 页头和码输入框采用签到态。
- 输入框边框、聚焦态、按钮和结果态提示保持同一色系。
- 成功结果态不使用大面积装饰，只突出动作完成与服务端时间。

### 6.5 签退页

- 与签到页共享结构，但切到签退态颜色。
- 让用户通过颜色也能确认当前不是签到页，降低误输概率。

### 6.6 工作人员管理页

- 作为本次增色收益最大的页面，动态码舱、Tabs、统计卡和批量操作区域统一走管理态。
- 动态码数字本身仍保持高对比深色，不因为背景着色而损失可读性。
- 一键签退等危险动作保留警示语义，不被管理态色彩覆盖。

### 6.7 名单页 / 个人中心

- 名单页沿用管理态，但比动态码页更克制，重点放在选中态和批量操作栏。
- 个人中心继续使用默认品牌层，不和业务态争抢层级。

## 7. 组件级改造

### 7.1 `MobilePage`

新增页面级 tone 概念，例如：

- `default`
- `brand`
- `checkin`
- `checkout`
- `staff`

职责：

- 控制 hero 区边框、高光、背景氛围
- 控制页面壳层可选的数据属性或类名
- 不直接决定正文区域的所有细节色，只提供页面上下文

### 7.2 `ActivityMetaPanel`

- 支持默认态与管理态的轻色阶摘要。
- 标题区、详情行、指标卡可根据 tone 使用不同边框和表面底色。
- 指标卡要避免四块都一样，可通过主次颜色让“应到 / 累计签到 / 已签退 / 未签退”层次更清楚。

### 7.3 `AppButton`

- 保留 `primary / secondary` 语义，但增加可选 color family。
- `primary` 根据页面 tone 使用不同业务态主色。
- `secondary` 保持克制，用轻着色描边或浅色背景承接业务态。

### 7.4 `CodeInput`

- 新增 tone，驱动外壳、输入框聚焦态、提示文字和按钮色彩。
- 仍保持单 input，不改交互模型。
- 目标是让六码输入区成为页面主视觉焦点，而不是普通表单的一部分。

### 7.5 `DynamicCodePanel`

- Tabs、动态码玻璃舱、统计组、刷新按钮统一纳入 `staff` tone。
- 签到码 / 签退码页签切换时，可在同一管理态内继续用青绿 / 橙色辅助区分动作。
- 不通过重动画制造差异，只用颜色和层次来强化当前动作。

### 7.6 `InlineNotice`、`StatusTag`、`AppBusinessNav`

- `InlineNotice`：保持语义主题，但补业务态容器风格。
- `StatusTag`：进行中 / 已完成的映射更明确，避免“都很灰”。
- `AppBusinessNav`：激活项增加更清楚的选中态，使全站导航不再像纯灰底标签。

## 8. 实施边界

以下内容不在本次设计范围：

- 改动接口、权限、路由与数据流
- 切换 `tdesign-mobile-react`
- 引入深色模式
- 引入新的插画、图标库或大面积视觉素材
- 追求展示型视觉特效

## 9. 可访问性与性能约束

- 正文与按钮文本必须满足基础对比度要求。
- 触控区维持至少 `44px`。
- 色彩只能增强动作识别，不能替代表意文案。
- `prefers-reduced-motion` 下不新增依赖位移的过渡。
- 不引入大面积实时模糊和高开销阴影，避免低端手机掉帧。

- 用户可以通过颜色和文案同时判断当前业务态。
- 共享组件改造后，现有业务流程和守卫逻辑不变。

## 11. 实施建议

建议按以下顺序落地：

1. 先改 `base.css`，补齐中性层、品牌层和业务态 token。
2. 再改 `MobilePage / AppButton / ActivityMetaPanel / CodeInput / DynamicCodePanel` 等共享层。
3. 最后把 tone 传到登录、列表、详情、签到、签退、管理、名单、个人中心页面，统一回归测试。

这样能用最少的结构改动，换取最大的全站视觉提升，并把风险控制在共享层可回归的范围内。
