# wxapp-checkin Web 响应式布局与管理员大屏展示设计

## 背景

当前 `wxapp-checkin/web` 的页面壳层是明显的手机优先实现，普通用户链路已经基本稳定，但仍存在两个直接影响使用的问题：

1. 手机页面面对不同尺寸、不同安全区形态的浏览器时，自适应边界不够稳，部分页面会出现留白失衡、组件挤压甚至横向溢出的情况。
2. 管理员展示动态 6 位码时，电脑浏览器仍沿用手机窄栏布局，导致动态码不够醒目，不适合在会场或教室里拿电脑面向人群展示签到或签退。

本设计只处理这两个问题，不扩展到新的业务能力、接口或路由体系。

## 目标

### 目标内

- 修复普通用户页面在不同手机宽度下的横向溢出与版面失衡问题。
- 保持普通用户页面在电脑浏览器中仍呈现为手机式窄栏中心布局，不额外做完整桌面化。
- 让管理员页在同一 URL 下根据浏览器宽度自动切换为电脑展示模式。
- 在管理员电脑展示模式下，把动态 6 位码放到页面中间最醒目的位置。
- 在管理员电脑展示模式下保留剩余时间与三个统计数字。
- 在管理员电脑展示模式下保留签到码/签退码切换与立即刷新能力，但降低其视觉优先级。

### 目标外

- 不新增后端接口。
- 不新增独立的大屏展示路由。
- 不把普通用户活动页、签到页、签退页改造成完整桌面后台布局。
- 不调整管理员页现有业务逻辑，例如动态码刷新、批量签退、活动详情拉取的触发方式。

## 用户已确认的决策

- 管理员动态码页采用“同一 URL 自动切换”方案，不引入新的独立页面入口。
- 管理员电脑模式采用“展示优先”的大屏样式，而不是控制台式并列布局。
- 管理员电脑模式同屏保留内容为“大码 + 剩余时间 + 三个统计数字”。
- 普通用户页在电脑浏览器中继续保持手机窄栏居中显示。
- 管理员电脑模式中保留切换签到码/签退码与立即刷新入口，但这些操作区只作为弱化的次级区域存在。

## 当前实现约束

### 现有结构

- 公共页面壳层集中在 `web/src/shared/ui/MobilePage.tsx` 与 `web/src/app/styles/base.css`。
- 管理员动态码展示集中在 `web/src/features/staff/components/DynamicCodePanel.tsx`。
- 管理员页面业务拼装集中在 `web/src/pages/staff-manage/StaffManagePage.tsx`。

### 当前问题根因

1. `MobilePage` 对所有业务页默认使用 `width: min(100%, 480px)` 的手机壳层策略，缺少更细的模式区分。
2. 现有通用样式没有系统性地处理超窄屏下的字距、左右留白、内部组件最小宽度和横向收缩边界。
3. `DynamicCodePanel` 在视觉上只定义了手机管理面板，没有“桌面展示态”的结构钩子和样式层。

## 方案对比

### 方案 A：在现有公共壳层和管理员面板上增加响应式模式

做法：

- 在 `MobilePage` 上增加轻量布局模式钩子。
- 普通用户页继续使用紧凑模式。
- 管理员页在同一组件树内根据断点切换到电脑展示态。

优点：

- 改动点集中，主要落在 `MobilePage`、`DynamicCodePanel`、`StaffManagePage`、`base.css`。
- 不需要复制一套管理员业务逻辑。
- 与现有组件结构最兼容，回归风险最低。

缺点：

- 公共壳层会比现在多一个布局维度，需要补少量测试保证类名钩子稳定。

### 方案 B：新增单独的响应式壳层组件

做法：

- 新增类似 `ResponsivePageFrame` 的壳层，再让页面分别接入。

优点：

- 语义更清楚，布局模式边界更显式。

缺点：

- 这次范围不大，引入额外抽象收益有限。
- 公共样式和页面接入点都会增加，成本高于必要值。

### 方案 C：为管理员页单独维护桌面版页面树

做法：

- 小屏和桌面各渲染一套不同的管理员内容结构。

优点：

- 视觉自由度最大。

缺点：

- 结构重复明显。
- 后续维护动态码、统计、刷新逻辑时容易出现两套 UI 分叉。

### 结论

采用方案 A。它最符合“同一 URL 自动切换、改动集中、尽量不分叉业务逻辑”的目标。

## 目标布局设计

### 1. 普通用户页布局策略

普通用户页包括但不限于：

- 活动列表页
- 活动详情页
- 签到页
- 签退页
- 结果页
- 个人资料页

这些页面继续保持手机式窄栏中心布局，即使在电脑浏览器中打开，也不扩展为后台式桌面页。

这样处理的原因：

- 普通用户的核心链路仍以手机输入、单手操作为主。
- 桌面上继续保持窄栏，可以避免页面在超宽视口中被拉散。
- 这次需求的重点不是给普通用户补桌面体验，而是把手机适配修稳，并给管理员增加展示能力。

### 2. 管理员页布局策略

管理员页仍然使用原有路由和数据加载逻辑，但布局行为按照断点切换：

- 小屏：保留现有管理页风格，继续以功能操作为主。
- 中屏过渡：保留单列结构，只放宽外层留白，不直接切成完整大屏展示态。
- 大屏：切换为“展示优先”的电脑模式，把动态码抬到视觉中心，统计下沉，操作弱化。

### 3. 管理员电脑展示模式的信息层级

一级信息：

- 当前动态 6 位码

二级信息：

- 剩余时间
- 当前是签到码还是签退码

三级信息：

- 签到人数
- 签退人数
- 未签退人数

四级信息：

- 切换签到码/签退码
- 立即刷新

该层级用于确保电脑模式首先服务“展示给别人看”，而不是优先服务“管理员自己在页面里密集操作”。

## 断点与响应式规则

### 断点定义

- `0px - 767px`：手机主场景
- `768px - 1023px`：过渡场景
- `1024px+`：管理员电脑展示场景

### 普通用户页规则

- 页面主壳层继续使用窄栏居中，但允许更细致地随着视口缩小收紧左右留白。
- 标题字号、卡片内边距、动态码输入字距在超窄屏下允许适度缩小，优先保证不横向溢出。
- 内部常见容器如卡片、Tabs、CellGroup、输入面板都要补 `min-width: 0` 或等价约束，避免子元素撑破父容器。
- 保留并校正 `safe-area` 相关 padding，确保刘海屏和底部手势区环境下内容与导航不被遮挡。

### 管理员电脑展示规则

- `MobilePage` 的桌面展示态不再锁死 `480px`。
- 电脑模式首屏内容以纵向三层结构为主：
  - 顶部：角色信息、活动标题、返回入口
  - 中部：超大动态码卡片
  - 底部：三个统计数字与弱化操作区
- 大号动态码在桌面断点单独放大字号和字距，确保远距离识别能力明显高于手机模式。
- 切换动作和刷新按钮保留，但收纳到次级区域，不与大码处于同一视觉竞争层。

## 组件设计

### MobilePage

新增轻量布局模式，例如：

- `compact`：普通手机优先壳层
- `showcase-auto`：在大屏上可切展示态的壳层

要求：

- 不改变现有 children 传入方式。
- 不引入新的业务状态。
- 通过稳定的 DOM 属性或类名输出布局模式，供 CSS 和测试使用。

### DynamicCodePanel

保留现有 props：

- `actionType`
- `codeSession`
- `loading`
- `onActionChange`
- `onRefresh`

新增职责：

- 输出桌面展示态所需的结构钩子。
- 在桌面展示态中重排“动态码、倒计时、统计、弱操作”的视觉顺序。

不新增职责：

- 不接管活动详情拉取。
- 不接管批量签退逻辑。
- 不接管页面级错误处理。

### StaffManagePage

继续负责：

- 活动详情加载
- 动态码加载与刷新
- 回前台重拉
- Wake Lock 请求
- 批量签退触发

新增内容仅限：

- 为 `MobilePage` 指定展示模式。
- 保持管理员页在不同断点下复用同一套数据源。

## 数据流与状态策略

### 保持单一数据源

管理员手机模式与管理员电脑模式必须继续复用同一份 `codeSession` 和 `detail` 数据。

原因：

- 避免同一页在不同断点下出现统计口径不一致。
- 避免为了布局切换复制请求逻辑。
- 保持动态码自动刷新、剩余时间计算和统计口径不变。

### 不新增布局状态机

断点切换由 CSS 主导，不新增额外的 React 布局状态。

原因：

- 浏览器宽度变化属于典型展示层能力。
- 当前需求不要求基于 JS 做复杂模式切换。
- 这样可以避免在断点附近出现模式闪动、状态重建或额外副作用。

## 异常与降级处理

- 当 `codeSession` 为空时，管理员电脑模式仍保留大码卡片骨架，内容展示为占位码 `------` 与加载文案，避免版面突然坍塌。
- 当动态码正在刷新时，只更新显示内容，不让桌面展示结构发生明显跳动。
- 当浏览器不支持 Wake Lock 时，仍沿用当前页面提示，不因为布局模式切换改变提示逻辑。
- 当浏览器位于 `768px - 1023px` 区间时，不强制进入完整展示模式，避免平板横屏体验处于展示和管理之间的尴尬状态。

## 测试与验证策略

### 自动化测试

至少补以下测试：

- `MobilePage` 能输出稳定的布局模式钩子。
- `DynamicCodePanel` 能输出稳定的管理员展示态结构钩子。

如果测试粒度允许，可进一步验证：

- 桌面展示态中仍然保留切换和刷新入口。
- 占位码 `------` 在无数据时仍可正常渲染。

### 命令验证

在 `wxapp-checkin/web` 内执行：

- `npm test`
- `npm run lint`
- `npm run build`

### 人工验收

重点检查以下宽度：

- `320px`
- `375px`
- `430px`
- `768px`
- `1024px`
- `1440px`

重点看：

- 是否仍存在横向滚动或内容溢出
- 普通用户页在桌面浏览器中是否仍保持手机式窄栏
- 管理员电脑模式的大码是否足够醒目
- 统计信息是否可读但不过度抢焦点
- 切换与刷新按钮是否仍然可用且视觉上确实退居次级

## 风险与控制

### 风险 1：普通用户页样式调整影响现有稳定页面

控制方式：

- 只在公共壳层和明确的问题样式上做收敛，不顺手重做整体视觉。
- 用测试和多断点人工检查覆盖签到/签退主链路。

### 风险 2：管理员桌面展示态和手机管理态样式互相污染

控制方式：

- 通过稳定布局模式钩子隔离桌面展示态样式。
- 不把桌面大屏样式直接写成全局无条件覆盖。

### 风险 3：断点边界行为不清晰

控制方式：

- 固定使用 `1024px+` 进入完整展示模式。
- `768px - 1023px` 明确只做过渡，不做大屏展示。

## 实施后的完成标准

满足以下条件才算完成：

1. 普通用户页面在主流手机宽度下不再出现横向溢出。
2. 普通用户页面在电脑浏览器中仍保持手机窄栏视觉。
3. 管理员页面在 `1024px+` 浏览器宽度下自动切为展示优先的大屏模式。
4. 电脑模式下动态 6 位码位于页面中部的明显位置。
5. 电脑模式下仍可看到倒计时与三个统计数字。
6. 电脑模式下仍保留切换签到码/签退码与立即刷新能力，但这些控件不抢视觉焦点。
7. 测试、lint、build 全部通过。
