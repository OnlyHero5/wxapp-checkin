# `wxapp-checkin/web` 全站视觉重设计方案

更新日期：2026-03-28  
适用范围：`wxapp-checkin/web` 当前正式前端

## 1. 背景与目标

本轮改造直接由用户提供的 3 张问题截图触发，目标不是做局部补丁，而是把 `wxapp-checkin/web` 的整套移动端视觉体系统一重做。

当前问题已经不是单个组件“不够好看”，而是同时存在以下几类失衡：

- 页面壳层过淡，导致标题、分组、卡片、按钮之间缺少层级。
- 活动列表、活动详情、staff 管理页、名单页之间视觉语言不一致。
- 动态码区域存在错误图片 / 错乱装饰，说明当前展示方式存在高风险实现。
- 名单修正区的动作块与成员信息块布局冲突，出现明显塌陷。
- 业务 tone 虽然存在，但颜色使用偏弱、区分不足，导致“签到 / 签退 / staff / 普通浏览”语义不够清晰。

本轮重设计需要同时满足两条硬约束：

1. **组件优先**：能直接用 `tdesign-mobile-react` 的地方尽量用组件库，不新增重复手写表单、列表、弹层、按钮、单元格壳层。
2. **视觉显著升级**：页面必须明显比当前更有层次、更有色彩、更适合移动端值守与浏览，不接受“只是把旧样式稍微加深一点”的伪改版。

## 2. 改造范围

本轮覆盖 `wxapp-checkin/web` 全部正式页面与会被这些页面复用的共享 UI 壳层：

- `LoginPage`
- `ActivitiesPage`
- `ActivityDetailPage`
- `CheckinPage`
- `CheckoutPage`
- `StaffManagePage`
- `ActivityRosterPage`
- `ProfilePage`
- 共享壳层与共享视觉组件：
  - `MobilePage`
  - `ActivityMetaPanel`
  - `AppButton`
  - `InlineNotice`
  - `StatusTag`
  - `AppBusinessNav`
  - 相关全局 token / 页面布局样式

不在本轮范围内：

- 后端接口、业务鉴权、字段协议调整
- `suda_union/`、`suda-gs-ams/` 任意代码修改
- 抛弃当前组件库另起一套 UI 框架

## 3. 设计原则

### 3.1 视觉方向

整体采用“**清亮、鲜明、移动优先的活动工作台**”方向：

- 保持浅底，避免做成压抑的深色 SaaS。
- 强化首屏标题、状态和操作的对比，不再让整页发灰发白。
- 页面像“可执行的业务面板”，不是“散开的白卡片列表”。
- 色彩变多，但必须是**有业务语义的多色**，不是随机堆颜色。

### 3.2 组件策略

优先复用以下组件库能力，不再回退为自制壳层：

- 表单：`Form`、`FormItem`、`Input`、`Search`
- 列表与单元格：`List`、`CellGroup`、`Cell`
- 页签：`Tabs`、`TabPanel`
- 操作确认：`ActionSheet.show`、`Dialog.confirm`
- 反馈：`Skeleton`、`Badge`、`CountDown`
- 交互：`SwipeCell`、`Checkbox`、`Navbar`

允许保留的少量定制，只限于组件库没有直接覆盖且截图已证明必须自定义的场景：

- `MobilePage` 页面壳层
- 动态码大号展示块
- 活动信息组合面板的字段编排
- 业务状态色与按钮 tone 的映射

### 3.3 少手写规则

本轮禁止新增以下类型的重复轮子：

- 手写新的“通用卡片组件库”
- 手写新的 Tabs / Dialog / ActionSheet / Checkbox / List 行为
- 通过复杂 DOM 嵌套去伪装已经有组件库可承担的结构

允许的手写样式只做三件事：

1. 定义设计 token；
2. 修复组件库组合后的业务布局；
3. 补足组件库不覆盖的大字展示、页头气质和业务语义装饰。

## 4. 视觉系统方案

### 4.1 新 token 方向

保留 CSS 变量体系，但把当前偏灰的配色改成更有区分度的业务色板：

- 页面背景：暖白蓝灰渐层，增强空间感。
- 基础 surface：高对比白卡 + 更稳定的边框色。
- 品牌主色：更干净的亮蓝，用于普通浏览与主导航。
- staff tone：更明确的紫蓝，用于管理态区分。
- 签到 tone：青绿，用于“已签到 / 签到码 / 成功动作”。
- 签退 tone：橙金，用于“签退码 / 已签退 / 离场动作”。
- 警告与错误：暖红与琥珀，不再与普通文案混色。

### 4.2 版式层级

页面统一调整为四层视觉结构：

1. **强页头层**：标题、描述、关键操作入口，成为首屏锚点；
2. **重点业务块**：活动主卡、动态码主卡、签到操作主卡；
3. **信息块层**：活动信息、统计、说明、名单；
4. **辅助操作层**：次级按钮、返回链路、补充说明。

这样处理后，用户进入任意页面时，都能先看到“这页最重要的东西”，而不是被一堆同权重白块淹没。

### 4.3 圆角、阴影与边框

遵循“正常、克制、但比现在更精致”的标准：

- 不做夸张 24px 以上圆角泛滥。
- 卡片圆角收敛到统一尺度，按钮圆角略小于卡片。
- 阴影改成短而稳的层级阴影，不做漂浮感过强的大投影。
- 边框颜色更明确，用边框 + 阴影共同建立层级，不靠一味加深背景。

## 5. 页面与组件改造方案

### 5.1 `MobilePage`

`MobilePage` 从“弱壳层容器”升级为全站统一页面骨架：

- 统一页头区的间距、标题字号、描述宽度、操作对齐。
- 增加页面 tone 到壳层的视觉投影，让普通页、签到页、staff 页能有稳定气质差异。
- 内容区统一 section gap、卡片边距与安全区留白。
- 保留 `Navbar`，不手写第二套导航。

### 5.2 活动列表 / 活动详情

活动列表和详情页统一成“活动主卡 + 关键信息 + 明确动作入口”的结构：

- 活动标题、类型、时间地点、状态标签形成首组强层级。
- “我的状态 / 报名状态 / 是否签到 / 是否签退”改成更容易扫读的状态信息。
- staff 入口不再和普通动作混成同一层级，而是作为明确的管理入口动作。
- 搜索框、Tabs、列表卡片的间距和卡片边界统一。

### 5.3 登录 / 签到 / 签退

这些动作页统一成“短流程高聚焦”页面：

- 首屏只保留当前动作所需信息，避免过多无关块打断操作。
- 登录页强化品牌识别、输入区与提交按钮的主次关系。
- 签到 / 签退页强化动态码输入区与提交反馈区，不再让辅助文案夺走注意力。
- 表单仍使用组件库 `Form` / `Input`，不手写输入壳。

### 5.4 staff 管理页动态码区

这是本轮重点修复点。

现状问题：

- 当前 `DynamicCodeHero` 直接使用 `Badge shape="ribbon-left"` 包裹整个主卡，截图已经显示这里存在错误图片/错乱视觉。
- 动态码区的标签、码值、倒计时、统计和刷新动作之间缺少稳定的构图关系。

重做方案：

- 去掉当前 `ribbon-left` 包裹方式，改为**自有标题条 + 码值主显示区 + 倒计时副信息区**。
- `Tabs` 继续使用 TDesign，不重写页签行为。
- `CountDown`、`Skeleton` 继续使用 TDesign，不重写倒计时与骨架能力。
- 主卡中让码值成为绝对视觉中心，动作类型只做明确标签，不再挤占卡面。
- 统计卡与刷新按钮从主卡中拆出稳定层级，避免全部堆在一个视觉块里。

### 5.5 名单页 / 批量修正

这是截图中第二个重灾区。

现状问题：

- `SwipeCell` 的业务动作块与名单主内容宽度失衡。
- 修正操作块被拉成长竖条，和名单信息区互相争宽。
- 用户无法一眼看清“成员信息”和“我现在要执行什么动作”。

重做方案：

- 保留 `SwipeCell` 作为行内动作能力，不重写滑动交互。
- 重新定义动作块宽度、最小高度、排版方式和文案层级，防止纵向拉爆。
- 名单主卡改成更清晰的“姓名/学号/选择/签到状态/签退状态”扫描布局。
- 批量修正区变成更明确的 summary + 主操作按钮，不再像一块空白占位面板。

### 5.6 底部导航与个人页

- 底部导航与页面色调一起统一，强化当前页态。
- 个人页改成更简洁但更完整的账户面板，保留查看信息与退出操作。
- 保证全站“主按钮 / 次按钮 / 文本链接 / 状态标签”一致。

## 6. 工程实现策略

### 6.1 优先修改顺序

按影响面从大到小推进：

1. 设计 token 与全局页面壳层
2. 共享 UI 组件（按钮、状态、信息面板、导航）
3. 活动列表 / 活动详情
4. staff 管理页动态码区
5. 名单页与批量修正区
6. 登录 / 签到 / 签退 / 我的页

### 6.2 文件组织策略

优先复用现有文件与共享组件，不新增碎片化样式文件泛滥。

如需拆分样式，遵循：

- 页面壳层与 token 继续集中在 `src/app/styles`
- 业务页专项样式留在现有 staff / page shell 体系下
- 不为单个微小视觉点新建大量孤立文件

### 6.3 注释策略

涉及 `wxapp-checkin/` 的改动，新增中文维护注释重点说明：

- 为什么这里保留组件库、为什么那里必须自定义；
- 动态码主卡为什么不用现有 `Badge ribbon`；
- 名单修正区为什么采用某种布局，避免后续再改回塌陷结构；
- 颜色与 tone 的业务语义边界。

## 7. 验证方案

实施后至少完成以下验证：

- `wxapp-checkin/web` 页面能正常构建与通过基础检查。
- 活动列表、活动详情、staff 管理页、名单页在浏览器里人工复核。
- 对照本次 `截图/1.png`、`截图/2.png`、`截图/3.png` 的问题逐项确认：
  - 动态码区不再出现错乱图片；
  - 名单修正区不再塌陷；
  - 活动列表与相关页面层级明显增强；
  - 色彩更丰富但仍保持业务语义清晰；
  - 没有因重构再次引入 URL submit、布局溢出或点击区域异常。

## 8. 风险与控制

主要风险：

- 过度追求视觉，重新引入手写壳层泛滥；
- 动态码区 DOM 调整后影响既有测试；
- 名单页布局重排后，窄屏兼容性再次出问题。

控制策略：

- 任何交互性能力优先交给组件库；
- 只改视觉结构，不重写业务 hook；
- 每完成一块就跑对应页面测试 / 构建验证；
- 所有新增自定义 DOM 都必须能回答“组件库为什么不能直接做这件事”。

## 9. 最终结果标准

实施完成后，`wxapp-checkin/web` 应达到以下状态：

- 全站页面像同一个产品，而不是若干临时页拼接。
- 截图中指出的错图、布局塌陷、组件打架问题全部消失。
- 页面颜色更丰富，且每种颜色都服务于明确业务语义。
- 组件库使用比例提高，没有回退到大面积手写表单/列表/弹层。
- staff 值守页面一眼能看清当前码、当前状态、当前可执行动作。
