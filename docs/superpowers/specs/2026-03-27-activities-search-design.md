# wxapp-checkin 活动列表搜索与分页联动设计

## 背景

当前 `wxapp-checkin/web` 的活动列表页已经支持分页加载，但仍然缺少搜索能力。

这在平时问题不明显，但在每年 11 月左右奖学金评审阶段，会集中出现一批“补次数 / 漏加分”的历史活动修正需求。当前工作流是：

1. 学生先在 PC 系统补报名。
2. 工作人员在 PC 系统审核通过。
3. 工作人员回到手机 Web，在活动管理链路里修正该学生的签到 / 签退状态。

当前列表页会把当前用户可见活动按时间倒序列出。对于工作人员来说，久远活动只能不断翻页查找；对普通用户来说，如果自己参加过的活动较多，也同样缺少快速定位手段。与此同时，PC 系统当前已有 430+ 个活动，如果为了“方便搜索”而一次性把所有活动拉到前端再本地过滤，会直接放大数据库读取、接口传输和移动端渲染压力。

本设计只解决“活动列表页搜索 + 分页联动 + 后端读取优化”这组问题，不扩展到高级筛选、年份筛选或新的独立搜索页。

## 目标

### 目标内

- 在手机 Web 活动列表页增加搜索框。
- 普通用户和工作人员都可以使用搜索。
- 搜索与现有分页能力联动，避免为了搜索而一次性读取全量活动。
- 搜索范围覆盖：
  - 活动标题
  - 活动地点
  - 活动描述
  - 数据库原始活动 ID
  - Web 前端暴露的活动 ID，例如 `legacy_act_123`
- 搜索结果继续遵守当前权限边界：
  - 普通用户只能搜到自己本来就可见的活动
  - 工作人员可以搜到全部活动
- 搜索结果继续沿用当前“正在进行 / 已完成”分组展示。
- 顺手消除当前活动列表服务中的 N+1 用户状态读取问题，减少每页列表对数据库的额外压力。

### 目标外

- 不新增新的活动列表路由。
- 不新增独立搜索页或高级筛选抽屉。
- 不新增年份、类型、状态等多条件筛选。
- 不改变活动详情、签到、签退、工作人员管理页的业务规则。
- 不引入全文检索引擎、搜索投影表或缓存层。

## 用户已确认的决策

- 搜索框不仅工作人员可用，普通用户列表页也要提供。
- 搜索字段不只搜标题，还要同时覆盖地点、描述和活动 ID。
- 搜索必须与分页结合，不能通过前端一次性拉全量数据来实现。

## 当前实现与约束

### 当前结构

- 前端活动列表页集中在 `web/src/pages/activities/ActivitiesPage.tsx`。
- 前端活动列表接口封装集中在 `web/src/features/activities/api.ts`。
- 后端列表接口是 `GET /api/web/activities`。
- 后端活动列表服务集中在 `backend-rust/src/service/activity_service.rs`。
- 后端活动列表 SQL 集中在 `backend-rust/src/db/activity_repo.rs`。

### 当前问题

1. 列表页只有分页，没有搜索，久远活动查找成本高。
2. 若直接改成“前端全量拉取后本地搜索”，会把 430+ 活动一次性传给移动端，违背当前分页设计初衷。
3. 当前后端列表服务在拿到分页活动后，会对每条活动逐条调用 `find_user_activity` 读取当前用户状态，存在明显 N+1 查询问题。

## 方案对比

### 方案 A：前端只对已加载列表做本地搜索

做法：

- 保持后端接口不变。
- 前端只在已经加载到页面的活动里做本地过滤。

优点：

- 改动最小。

缺点：

- 只能搜到“已经翻到前端的数据”，无法解决久远活动查找问题。
- 用户仍然可能为了搜历史活动不断点“加载更多”，实际压力并没有消失。
- 与本次“避免全量读取”的核心目标矛盾。

### 方案 B：在现有分页接口上增加服务端搜索

做法：

- 继续复用 `GET /api/web/activities`。
- 新增可选查询参数 `keyword`。
- 服务端先按权限边界过滤，再按关键字过滤，最后分页返回。

优点：

- 与当前分页接口和页面结构最兼容。
- 不新增路由，不分叉普通用户与工作人员列表页。
- 能真正把搜索成本留在服务端分页范围内。

缺点：

- 需要同步修改前端、后端和测试。

### 方案 C：单独新增搜索页或高级筛选能力

做法：

- 新增独立搜索页，或在列表页上增加年份、状态、类型等组合筛选。

优点：

- 扩展性强。

缺点：

- 远超当前需求。
- 接口和交互都会明显膨胀。
- 会拖慢当前问题的交付节奏。

### 结论

采用方案 B：在现有分页接口上增加服务端搜索，并顺手修复当前列表服务的 N+1 查询问题。

## 总体设计

### 1. 交互设计

- 搜索框放在活动列表页顶部，位于分组 Tabs 之前。
- 普通用户和工作人员都显示同一套搜索入口。
- 搜索框使用当前已安装的 `tdesign-mobile-react` `Search` 组件，不手写输入框壳层。
- 搜索采用“显式提交”模式，而不是输入即查：
  - 点击搜索
  - 手机键盘提交
  - 清空搜索
- 提交搜索后，列表重置到第一页，并按新关键字重新加载。
- 点击“加载更多”时，必须沿用当前关键字继续翻页。
- 清空关键字后，恢复默认分页列表。

采用显式提交的原因：

- 可以减少移动端输入过程中的频繁请求。
- 与当前“加载更多”按钮式分页模型更一致。
- 能避免在弱网或高并发输入场景下制造无意义的请求风暴。

### 2. 权限与可见性边界

- 搜索只缩小当前用户本来就有权看到的活动集合，不扩大权限。
- 工作人员搜索范围是全部活动。
- 普通用户搜索范围是“自己本来就可见的活动”，也就是与当前列表页一致：
  - 已报名
  - 已签到
  - 已签退
- 普通用户不能通过猜活动标题或 ID 搜出本来无权看到的活动。

### 3. 搜索范围

关键字统一匹配以下字段：

- `suda_activity.name`
- `suda_activity.location`
- `suda_activity.description`
- `suda_activity.id`
- `CONCAT('legacy_act_', suda_activity.id)`

这样做的原因：

- 工作人员经常记得的是活动名关键词。
- 某些历史活动更容易通过地点或描述中的特征词定位。
- 处理补录 / 漏加分时，也可能直接拿到活动编号或前端活动 ID。

## API 设计

### 现有接口

- `GET /api/web/activities`

### 变更后接口

- `GET /api/web/activities?page=1&page_size=50&keyword=奖学金`

### Query 参数

- `page`
- `page_size`
- `keyword`

### `keyword` 规则

- 可选参数。
- 服务端先做 `trim`。
- `trim` 后为空字符串时，按“未传关键字”处理。
- 最大长度限制为 `100` 个字符。
- 超长时返回 `invalid_param` 业务错误。

### 兼容性

- 老前端如果不传 `keyword`，接口行为与当前一致。
- 不新增新端点，只扩展现有列表接口的查询能力。

## 后端设计

### 1. API 层

`backend-rust/src/api/activity.rs`

- 在 `ActivityListQuery` 中新增：
  - `keyword: Option<String>`
- 把 `keyword` 继续传给 `activity_service::list_activities(...)`。

### 2. Service 层

`backend-rust/src/service/activity_service.rs`

新增职责：

- 归一化 `keyword`
- 校验长度
- 把搜索参数透传到仓储层

建议新增辅助函数：

- `normalize_keyword(value: Option<String>) -> Result<Option<String>, AppError>`
- `build_activity_state_map(...)` 或等价命名的批量状态收口函数

行为要求：

- 空关键字与未传关键字视为同一语义。
- 有关键字时，分页逻辑不变，仍然使用 `page_size + 1` 判断 `has_more`。
- 列表结果仍然沿用当前 DTO，不新增额外展示字段。

### 3. 仓储层

`backend-rust/src/db/activity_repo.rs`

#### Staff 列表

- 在现有 `list_staff_activities` 基础上增加 `keyword` 参数。
- 当 `keyword` 为空时，行为与当前一致。
- 当 `keyword` 有值时，在 `suda_activity` 维度增加搜索条件后再做聚合与分页。

#### 普通用户列表

- 保留现有 `EXISTS` 可见性约束。
- 在同一个查询中叠加关键字过滤。
- 先保证“用户有权看见该活动”，再判断关键字是否匹配。

#### LIKE 匹配

- 文本字段使用 `%keyword%` 模糊匹配。
- 为避免用户输入 `%`、`_` 被误解释为通配符，必须先做 LIKE 转义。
- `legacy_act_123` 匹配中包含 `_`，因此如果不转义，用户输入会产生错误命中范围。

建议新增仓储辅助函数：

- `escape_like_keyword(value: &str) -> String`
- `build_keyword_pattern(value: &str) -> String`

#### ID 匹配

- 数字 ID 和 `legacy_act_` 格式 ID 都参与模糊匹配。
- 不要求专门解析成“精确 ID 搜索”分支，先统一纳入现有关键字匹配口径即可。

### 4. N+1 查询优化

当前列表服务会对每个活动逐条查询用户状态，这在分页和搜索同时存在时会继续放大数据库压力。

因此本次设计要求顺手新增“按活动 ID 批量读取某个用户状态”的仓储接口，例如：

- 输入：
  - `student_id`
  - `legacy_activity_ids[]`
- 输出：
  - `activity_id -> UserActivityRow` 的映射

这样每页列表的读取成本从：

- 1 次活动列表查询 + N 次用户状态查询

收口为：

- 1 次活动列表查询 + 1 次批量用户状态查询

这样做不改变前端契约，但能把这次“搜索 + 分页”的性能收益真正落到数据库层。

## 前端设计

### 1. API 封装

`web/src/features/activities/api.ts`

`getActivities` 入参扩展为：

- `page?: number`
- `page_size?: number`
- `keyword?: string`

要求：

- 只有 `keyword` 非空时才拼到 query string。
- 统一通过 `URLSearchParams` 组装，避免手写字符串拼接出错。

### 2. 页面状态

`web/src/pages/activities/ActivitiesPage.tsx`

建议新增两份状态：

- `draftKeyword`
  - 输入框当前值
- `keyword`
  - 已提交、真正参与请求的值

这样做的原因：

- 用户输入时不立即触发请求。
- 页面只在“用户明确提交”后切换请求条件。

### 3. 页面行为

#### 首次进入

- 以空关键字拉取第一页。

#### 提交搜索

- 对输入做 `trim`。
- 把 `keyword` 更新为提交值。
- 清空旧错误。
- 重置分页状态：
  - `page = 1`
  - `hasMore = false`
  - `activities = []`
- 请求新的第一页结果。

#### 加载更多

- 使用当前 `keyword` 请求下一页。
- 继续沿用现有按 `activity_id` 去重合并逻辑。

#### 清空搜索

- 同时清空 `draftKeyword` 和 `keyword`。
- 回到默认第一页列表。

### 4. 页面展示

- 搜索结果回来后，继续通过当前 `groupVisibleActivities(...)` 做分组展示。
- 普通用户仍然把完成分组文案显示为“历史活动”。
- 搜索为空且结果为空时，保持现有空态语义。
- 搜索非空且结果为空时，建议沿用统一空态组件，但文案切为更明确的“未找到匹配活动”。

本次设计不要求单独新增“搜索结果页”或“搜索中状态页”。

## 错误处理

- 搜索请求失败时：
  - 保留输入框当前内容
  - 保留当前已提交关键字
  - 继续用现有 `InlineNotice + 重新加载`
- `SessionExpiredError` 行为保持不变，仍然跳转登录页。
- `keyword` 非法时，直接展示后端返回的业务错误文案，不在前端额外猜测。

## 测试设计

### 前端

至少补以下测试：

1. `getActivities` 在有关键字时能正确拼出 `keyword` 查询参数。
2. 提交搜索后，列表会以 `page=1 + keyword` 重新请求。
3. 搜索后点击“加载更多”会继续沿用同一个 `keyword`。
4. 清空搜索后会恢复默认列表请求。
5. 搜索无结果时，页面能展示稳定空态。

### 后端

至少补以下测试：

1. `keyword` 归一化测试：
  - `None`
  - 空白字符串
  - 正常关键字
  - 超长关键字
2. LIKE 转义辅助函数测试：
  - `%`
  - `_`
  - `legacy_act_123`
3. 批量用户状态映射逻辑测试：
  - 空活动列表
  - 单条活动
  - 多条活动

如果当前仓储层缺少可直接跑数据库级别测试的基础设施，本轮至少要把关键的纯函数和 service 归一化逻辑测试补齐。

## 验收标准

- 活动列表页顶部存在可用搜索框。
- 普通用户和工作人员都可以搜索活动。
- 搜索可命中标题、地点、描述、数据库活动 ID、前端活动 ID。
- 搜索结果仍然分页返回，不会一次性拉取全量活动。
- 搜索结果仍然遵守普通用户 / 工作人员各自的权限边界。
- 搜索后点击“加载更多”不会丢失当前关键字。
- 清空搜索后恢复默认列表。
- 后端活动列表读取不再对每条活动逐条补查用户状态。

## 风险与后续

### 当前接受的风险

- `%keyword%` 模糊匹配对大型数据量的索引利用有限。
- 但以当前 430+ 活动规模，这个成本仍然可接受。
- 本轮先优先解决可用性问题和当前明显的 N+1 查询问题。

### 后续可选演进

若未来活动量继续明显增长，再评估：

- 活动搜索专用索引
- 结构化筛选条件
- 搜索投影表
- 前缀匹配或更精细的 ID 快速定位分支

这些都不是本轮设计范围。
