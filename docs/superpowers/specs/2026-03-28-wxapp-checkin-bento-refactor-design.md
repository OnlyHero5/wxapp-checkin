# `wxapp-checkin/web` 全站便当布局与名单状态自愈重构设计

更新日期：2026-03-28  
适用范围：`wxapp-checkin/web`

## 1. 背景

本轮改造由两个同时存在的问题触发：

1. 当前前端整体视觉碎片化，尤其是活动列表中“一个活动被拆成三张卡”的实现，与移动端浏览习惯冲突。
2. staff 名单管理链路存在脏数据与动作口径错位：部分成员会出现“已签退但未签到”的非法组合，导致批量签退、批量取消签退等操作被后端拒绝。

本轮目标不是只修一处页面，而是同时完成：

- 全站正式页面统一为同一套“纵向信息带型便当布局”；
- 活动列表回归“一活动一主卡”；
- 名单页与管理页接入异常状态自动自愈；
- 单人修正、批量修正、批量签退围绕同一套状态机和动作命令工作。

## 2. 用户已确认的关键决策

### 2.1 布局方向

用户明确要求全站正式页面都采用便当布局，最终选择的风格方向为：

- **纵向信息带型便当布局**
- 保持移动端单列稳定性
- 允许有明显块面分区，但不走高风险棋盘拼块

### 2.2 名单异常口径

用户明确要求：

- 任何情况下都不允许出现“已签退但未签到”的状态。
- 如果执行“设为已签退”，且成员当前未签到，则必须一并补成已签到。
- 如果后端接口返回 `checked_out = true && checked_in = false`，前端不能只兼容展示，而要主动通知后端自愈。
- 异常自愈要同时发生在：
  - `活动管理页`
  - `参会名单页`

### 2.3 名单交互方式

用户要求保留 `SwipeCell` 单人滑动动作，不改成纯按钮式列表。

## 3. 现状根因

### 3.1 活动列表为何会出现“一活动三卡”

当前 `ActivityMetaPanel` 会把同一活动拆成三个 `CellGroup`：

- `summary`
- `detail`
- `metric`

活动卡片只是把这三个块包在同一个容器里，并没有真正形成“一个活动一张卡”的结构。因此即便继续调颜色和边距，视觉上仍然会像三张独立卡片。

### 3.2 名单状态为何会触发后端非法组合

当前名单修正链路的问题分成两层：

1. 页面直接消费后端原始 `checked_in / checked_out`，没有先归一化异常状态。
2. 前端动作层和后端契约层口径不一致。后端实际接受的是四种单命令 patch，但前端名单语义容易被误解成“直接拼最终双布尔结果”。

结果是：

- 脏数据成员会被渲染成自相矛盾的展示态；
- 行内动作和批量动作容易给出错误语义；
- 批量取消签退等操作会在后端命中“组合非法”。

## 4. 设计目标

### 4.1 视觉目标

- 全站正式页面统一为同一套移动端便当骨架。
- 每个页面只允许一个最核心的主块抢占首屏视觉中心。
- 活动列表必须实现“一活动一主卡，卡内分区”。
- staff 页面、普通用户页面、签到页、签退页在统一骨架下保留清晰业务 tone 差异。

### 4.2 逻辑目标

- 前端不再展示、保留或继续放大“已签退但未签到”的非法状态。
- 管理页与名单页都能自动发现并自愈异常成员。
- 单人修正、批量修正、批量签退围绕同一套规范状态机执行。
- 任何批量链路都不能再因为前端错误状态理解而发出非法 patch。

## 5. 方案对比与最终选择

### 5.1 方案一：最小补丁

- 仅修改样式，让三个活动块看起来像一个外层卡片；
- 名单页只补异常展示与少量 patch 修正。

优点：

- 改动快。

缺点：

- 结构问题仍然存在；
- 无法支撑“全站便当布局”；
- 后续还会继续在旧骨架上打补丁。

### 5.2 方案二：结构化重构

- 重写共享骨架；
- 新增名单规范状态 view-model；
- 修正页面动作与后端命令口径。

优点：

- 视觉与逻辑一起收口；
- 风险可控。

缺点：

- 相比最小补丁改动更大。

### 5.3 方案三：重骨架 + 重组件 + 重状态口径

- 在方案二基础上，进一步统一全站正式页面的视觉语义、主块节奏和动作带结构；
- 允许更完整的共享组件重组，而不是只改单页。

优点：

- 最符合用户对“全局都使用便当布局”的要求；
- 可以一次性收口活动列表、详情、管理、名单、登录、签到、签退、我的页的产品观感；
- 能把脏数据自愈机制直接嵌入 staff 核心工作流。

缺点：

- 改动面最大；
- 需要更严格的测试和构建验证。

最终选择：**方案三**。

## 6. 全站便当布局骨架

### 6.1 统一页面层级

所有正式页面统一采用四层结构：

1. **页面首块**
   - 标题
   - 页面说明
   - 返回/切换等关键入口
2. **主便当块**
   - 当前页面最重要的信息或操作核心
3. **辅助便当块**
   - 统计、说明、次级信息
4. **动作带**
   - 提交、管理、批量处理、危险操作等高频动作

### 6.2 页面首块原则

- 首块必须成为页面锚点，不能只是弱化标题容器。
- 不再堆叠多段同级浅色白块。
- tone 通过页头背景、强调色、动作层级体现，但不做高噪音装饰。

### 6.3 主块原则

- 每个页面只有一个主块可以成为首屏核心。
- 其它信息必须降级成辅助块，避免主次失衡。
- 页面从“很多卡片”切换为“一个主块 + 若干从块”。

## 7. 共享组件重构方案

### 7.1 `MobilePage`

`MobilePage` 升级为真正的全站页面骨架：

- 统一首块间距、标题尺寸、描述宽度与 header action 对齐方式；
- 内容区维持单列信息带，控制每个便当块的节奏；
- 普通、staff、签到、签退 tone 保留，但统一在同一骨架内表达。

### 7.2 `ActivityMetaPanel`

`ActivityMetaPanel` 从“三组 `CellGroup` 组合器”改为“单容器多分区主卡”：

- 标题区：活动名、类型、进度标签
- 状态区：我的状态或 staff 统计
- 信息区：时间、地点、说明
- 动作区：查看详情、进入管理、去签到、去签退

重点约束：

- 一个活动只允许渲染一个外层主卡；
- 分区只能存在于主卡内部；
- 不再输出 `summary / detail / metric` 三张视觉卡。

### 7.3 状态与动作组件

- `StatusTag` 保留，但统一服务于主卡内部状态区；
- `AppButton` 与文本动作统一收口到动作带语义；
- 名单页与管理页新增自有状态提示，但不另起通用组件库。

## 8. 页面级改造方案

### 8.1 `ActivitiesPage`

- 搜索框与 tabs 继续保留；
- 列表区域改为真正的便当列表；
- 每个活动只渲染一个主卡；
- 普通用户与 staff 共用同一主卡结构，只在动作带与状态区显示不同内容；
- 历史活动沿用同一骨架，但降低视觉权重。

### 8.2 `ActivityDetailPage`

调整为：

- 活动主卡
- 主要动作带
- 补充说明块

首屏直接回答：

- 这是什么活动；
- 我的当前签到/签退状态；
- 我现在能不能签到或签退；
- staff 是否可进入管理或名单页。

### 8.3 `StaffManagePage`

调整为“动态码工作台”：

- 活动摘要压成次级块；
- 动态码主卡成为唯一首屏核心；
- `Tabs` 继续保留签到码 / 签退码切换；
- 统计块与危险动作块从主卡中拆出；
- 页面进入后先执行异常状态扫描与自愈，再开放完整 staff 操作链路。

### 8.4 `ActivityRosterPage`

调整为：

- 活动摘要卡
- 批量修正卡
- 名单信息带

名单行规则：

- 保留 `SwipeCell`
- 每行一个成员卡
- 卡内展示姓名、学号、规范状态、时间、选择控件
- 滑动动作只承担“单人修正”，不承担状态解释

### 8.5 `LoginPage` / `CheckinPage` / `CheckoutPage` / `ProfilePage`

这些页面也统一进入便当布局：

- 登录页：账号输入主卡 + 提交动作带
- 签到页 / 签退页：活动摘要小卡 + 验码主卡 + 结果块
- 我的页：账户信息卡 + 会话/退出动作卡

## 9. 名单规范状态机

### 9.1 规范状态定义

前端不再直接让页面依赖原始布尔位，而是先归一成规范状态：

- `not_checked`
- `checked_in`
- `checked_out`

另外保留：

- `is_data_anomalous`

用于标记后端返回的异常原始值：

- `checked_out = true && checked_in = false`

### 9.2 强不变量

全站前端将以下规则视为硬约束：

- 只要是 `checked_out`，前端规范态就必须同时视为“已签到”。
- 前端展示态、动作态、选择态、批量态中，禁止出现“已签退但未签到”。

### 9.3 动作语义

单人和批量动作都只围绕四种命令展开：

- `set_checked_in`
- `clear_checked_in`
- `set_checked_out`
- `clear_checked_out`

其中：

- `set_checked_out` 的业务语义始终是“必要时自动补签到，再设为已签退”。
- `clear_checked_out` 的业务语义始终是“恢复为已签到未签退”。

## 10. 异常数据自动自愈

### 10.1 触发时机

以下两个页面都要自动执行异常扫描与自愈：

- `活动管理页`
- `参会名单页`

### 10.2 异常识别

命中以下条件即视为异常成员：

- `checked_out = true`
- `checked_in = false`

### 10.3 自愈命令

自愈请求不能发 `checked_in: true`，因为这会把成员修成“已签到未签退”，而不是“已签退”。

正确做法是：

- 调用 `attendance-adjustments`
- 发出 `set_checked_out` 语义命令
- 让后端把最终状态落成合法的 `(check_in = 1, check_out = 1)`

### 10.4 页面行为

页面加载流程改为：

1. 拉取页面所需数据；
2. 扫描异常成员；
3. 如果存在异常成员，先自动自愈；
4. 自愈成功后重新拉取最新数据；
5. 再展示正式名单、批量栏与完整操作入口。

### 10.5 自愈失败处理

如果自动自愈失败：

- 页面展示明确错误提示；
- 临时禁用受影响批量链路；
- 不再继续基于脏状态执行高风险操作。

## 11. 前端与后端契约收口

### 11.1 当前后端约束

Rust 后端当前对 `attendance-adjustments` 的约束是：

- 只接受四种单命令 patch；
- 非法组合会返回 `patch 组合非法`。

因此前端必须严格避免继续表达“双布尔最终态”的思路。

### 11.2 前端收口原则

- 页面层不拼 patch；
- 动作条不拼 patch；
- 单人修正与批量修正都只传动作枚举；
- 由统一 resolver 负责把动作枚举映射为后端可接受的单命令请求体。

## 12. 测试策略

### 12.1 先补失败测试，再改实现

本轮先新增或重写失败测试，覆盖：

- 活动卡回归单一主卡结构；
- 异常名单成员会被识别；
- 管理页与名单页都会触发异常自愈；
- 自愈请求使用 `set_checked_out` 语义，而不是错误双布尔 patch；
- 单人滑动修正、批量设签退、批量取消签退在异常数据修复后能继续工作。

### 12.2 页面与共享层验证

除单测外，还需验证：

- 全站页面骨架统一；
- 活动列表不再出现“一活动三卡”；
- `SwipeCell` 保留且动作区布局不再塌陷；
- 便当布局在手机窄屏下不溢出。

## 13. 实施顺序

按以下顺序推进：

1. 新增/调整测试，先固定期望行为；
2. 重构名单规范状态、自愈逻辑、动作 resolver；
3. 重构共享骨架与共享视觉组件；
4. 逐页替换：
   - 活动列表
   - 活动详情
   - staff 管理页
   - 名单页
   - 登录 / 签到 / 签退 / 我的页
5. 跑完整体验证。

## 15. 风险与控制

### 15.1 风险

- 全站视觉重构可能引入样式回归；
- 自愈逻辑如果放错时机，可能导致页面闪烁或重复请求；
- `SwipeCell` 与新名单布局之间可能再出现窄屏挤压；
- 共享组件改动过大时，可能影响签到、签退、详情等非 staff 页面。

### 15.2 控制措施

- 先从测试锁住关键行为，再推进实现；
- 自愈逻辑集中在页面级 hook，不分散进多个组件；
- 共享组件只保留真正跨页的结构能力，不新增第二套通用组件库；
- 每完成一个阶段就跑对应页面测试与构建验证。

## 16. 完成标准

本轮完成后应达到以下结果：

- 全站正式页面统一采用纵向信息带型便当布局；
- 活动列表实现“一活动一主卡”；
- staff 管理页与名单页均能自动发现并修复“已签退但未签到”异常数据；
- 批量签退、批量取消签退、单人滑动修正不再因状态错位触发后端非法组合；
- 页面视觉、动作层级与状态语义明显优于当前版本。
